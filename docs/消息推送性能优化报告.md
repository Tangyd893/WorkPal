# 消息推送性能优化报告

## 优化目标

本轮优化围绕“2000 人大群发消息”场景，优先降低数据库写入放大和重复发送风险，并为后续真实压测保留可重复脚本。

## 已落地内容

- 新增 `testing/performance/群发消息压测.js`，可用 k6 构造 2000 虚拟用户群发消息。
- 消息发送接口支持 `idempotency_key` 和 `Idempotency-Key` 请求头。
- 后端以 `(conv_id, sender_id, idempotency_key)` 做幂等去重，并清理 5 分钟前的幂等键。
- `messages` 表新增 `tier` 字段，默认新消息为 `hot`。
- 新增会话、时间范围、冷热分层相关索引，历史消息查询可按 `start_at` / `end_at` 走读扩散路径。
- 搜索索引链路继续基于 outbox + Redis Streams 消费者组，避免发送请求同步阻塞搜索写入。

## 压测命令

```bash
k6 run testing/performance/群发消息压测.js \
  -e WORKPAL_GATEWAY=http://localhost:8080 \
  -e WORKPAL_CONV_ID=1 \
  -e WORKPAL_USERNAME=admin \
  -e WORKPAL_PASSWORD=admin123
```

## 关注指标

- P50 / P99 消息发送延迟
- `workpal_http_requests_total` 的发送接口 QPS
- `workpal_http_request_duration_seconds` 的 P95/P99
- PostgreSQL `messages` 写入量
- Redis Streams 积压与 dead letter 数量
- IM 服务 goroutine 数与 GC 暂停

## 后续真实压测步骤

1. 使用 Docker Compose 启动 PostgreSQL、Redis、MinIO、后端服务、Prometheus 和 Grafana。
2. 创建一个包含足够成员的群会话。
3. 执行 k6 脚本并记录基线。
4. 观察 Grafana 的服务概览和接口明细看板。
5. 根据数据库写入、P99 延迟和 Redis 积压继续调整批处理、索引和实例数。
