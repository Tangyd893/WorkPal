## 1. 音视频通话（WebRTC）

### 1.1 什么是 WebRTC

**WebRTC（Web Real-Time Communication）** 是一种让浏览器和 App 之间直接进行实时音视频通信的技术标准。不需要安装插件，不需要第三方软件，直接通过网页或 App 就能打视频电话。

**在你的平台里能做什么：**
- 私聊时发起视频通话（类似微信视频）
- 创建多人视频会议
- 屏幕共享（会议中分享自己屏幕）

### 1.2 WebRTC 工作原理

#### 第一步：媒体采集

```go
// 伪代码：浏览器侧采集摄像头和麦克风
navigator.mediaDevices.getUserMedia({ video: true, audio: true })
// 返回 MediaStream，包含视频轨道和音频轨道
```

采集后的原始视频数据会进入编码器压缩（否则 1 秒原始视频太大，网络传不了）。

#### 第二步：建立连接（核心难点）

两个人的浏览器要建立直接通信，面临一个大问题：**NAT（网络地址转换）**。

```
你的电脑  →  路由器（NAT）  →  公网
对方电脑  →  路由器（NAT）  →  公网

问题：路由器的公网 IP/端口是动态的，两人怎么知道对方的地址？
```

解决方案靠三个协议：

| 协议 | 全称 | 作用 | 比喻 |
|------|------|------|------|
| STUN | Session Traversal Utilities for NAT | 帮助双方找出自己的公网 IP/端口 | 帮你照镜子看自己长什么样 |
| TURN | Traversal Using Relays around NAT | 打不通 P2P 时，用中继服务器转发流量 | 快递送不到就放驿站 |
| ICE | Interactive Connectivity Establishment | 综合框架，协调 STUN/TURN 找到最优路径 | 制定最优路线 |

**连接建立流程：**
```
用户A                          用户B
   │                               │
   │────── STUN 请求 ──────────────→│  互相问："你外面长啥样"
   │←────── 公网IP:端口 ────────────│  互相答
   │
   │  尝试直连（P2P）                │
   │────── UDP 包直接发 ────────────→│  如果路由器支持，直连成功
   │                               │
   │  （直连失败则走 TURN 中转）       │
   │══════ TURN 服务器中转 ════════│  所有流量通过 coturn 服务器
```

#### 第三步：信令传输（SDP 交换）

在真正建立音视频连接之前，两人需要先"握手"，交换 **SDP（Session Description Protocol）**：

```
SDP 包含：
- 我要发什么格式的视频/音频（编码器是 H.264 还是 VP8）
- 我的候选连接地址（IP:端口）
```

**问题：SDP 怎么传？** 你的服务器！因为两人还没建立直接连接，所以通过你的后端服务器转发 SDP。

```go
// 服务器收到用户A的 SDP offer，转发给用户B
type SignalingMessage struct {
    Type      string `json:"type"`   // "offer" / "answer" / "candidate"
    FromUser  string `json:"from"`
    ToUser    string `json:"to"`
    SDP       string `json:"sdp,omitempty"`        // offer/answer
    Candidate string `json:"candidate,omitempty"`  // ICE 候选
}
```

#### 第四步：实际传输（RTP/RTCP）

连接建立后，音视频数据通过 **RTP（Real-time Transport Protocol）** 传输，同时 **RTCP（RTP Control Protocol）** 负责质量反馈（丢包率、延迟）。

### 1.3 coturn 服务器部署

coturn 是最流行的开源 STUN/TURN 服务器。

```yaml
# Docker 部署 coturn
services:
  coturn:
    image: instrumentisto/coturn
    ports:
      - "3478:3478/udp"   # STUN/TURN 端口
      - "3478:3478/tcp"
    environment:
      TURNSERVER_ENABLED: 1
      SECRET: "your_turn_secret"    # 生成 auth token 的密钥
      REALM: "workpal.example.com"
```

### 1.4 在你的平台里的实现

**时机：** 用户点击"视频通话"按钮时：

```
1. 发送方 → 服务器：发起通话请求（你的 WebSocket/后端 API）
2. 服务器 → 接收方：推送来电通知
3. 接收方 → 服务器：接受/拒绝
4. 双方开始交换 SDP（通过你的服务器转发）
5. 双方建立 WebRTC 连接
6. 通话建立
```

| 组件 | 选型 | 说明 |
|------|------|------|
| 前端 | 原生 WebRTC / vue-webrtc / android-webrtc | iOS/Android/Web 三端 |
| TURN 服务器 | coturn | 开源，Docker 部署 |
| 信令服务器 | 你的 Go 后端 | 复用现有的 WebSocket 连接 |

## 2. 文档协作编辑（Yjs/CRDT，OT 对比）

### 2.1 当前推荐：Yjs/CRDT

**Yjs** 是成熟的前端协作编辑 CRDT 框架，适合和 React、富文本编辑器、WebSocket 同步协议结合。它能处理多人并发编辑、离线编辑、重连同步等问题，工程落地成本比从零实现 OT 更低。

**在你的平台里能做什么：** 类似飞书文档、Google Docs，让多人同时编辑同一个文档、表格、思维导图，实时看到彼此的改动。

当前建议：
- 前端使用 Yjs + 编辑器绑定（如 TipTap/ProseMirror/Monaco 的 Yjs binding）
- 后端提供文档权限、房间鉴权、持久化和 WebSocket 同步通道
- 后续再评估是否将文档更新日志持久化到 PostgreSQL 或对象存储

### 2.2 OT 算法是什么

**OT（Operational Transformation）** 仍然是理解协作编辑冲突处理的重要算法。以下内容作为原理说明保留。

### 2.3 问题的本质：冲突

假设文档初始内容是 `"Hello"`，两个用户同时编辑：

```
用户A：在位置 5 后面插入 " World"
用户B：在位置 5 后面插入 "!"

两个操作都发生在同一个位置（索引 5），结果是什么？
```

如果服务器按顺序执行：
- 先执行A："Hello World"，再执行B："Hello World!" ✅
- 先执行B："Hello!"，再执行A："Hello World!" ❌（A 期望的）

**顺序不同，结果就不同。** OT 就是解决这个问题的。

### 2.4 OT 算法原理

OT 的核心思想：**不改变操作执行的顺序，而是变换操作本身。**

```
用户A的操作：insert(" World", pos=5)
用户B的操作：insert("!", pos=5)

服务器收到两个操作，先执行哪个？
- 如果先执行 A → transform(B) → B' = insert("!", pos=5+len(" World"))
                              → 结果："Hello World!"
- 如果先执行 B → transform(A) → A' = insert(" World", pos=5+1)
                              → 结果："Hello World!"
```

无论哪种顺序，最终都得到 **"Hello World!"**——结果一致。

### 2.5 三种基本操作

OT 定义了三种原子操作：

```go
type Op struct {
    Type   string  // "insert" | "delete" | "retain"
    Pos    int     // 操作发生的位置
    Text   string  // insert 时使用，插入的文本
    Length int     // delete/retain 时使用，影响的字符数
}

// 示例
insert("hello", 5)   // 在位置 5 插入 "hello"
delete(5, 3)        // 删除位置 5 开始的 3 个字符
retain(10)           // 跳过 10 个字符（用于定位）
```

### 2.6 服务端架构

```
客户端 A ──┐
客户端 B ──┼──→ WebSocket ──→ 服务端 ──→ 操作转换引擎 ──→ 广播给所有客户端
客户端 C ──┘                            ↓
                                      持久化（Append-only log）
```

**Append-only log（只追加日志）：**
服务器按顺序记录所有操作，每个操作带一个递增的版本号。客户端拉取落后太多时，可以重放日志追到最新。

```go
type DocOperation struct {
    Version  int64
    UserID   string
    Op       Op
    Timestamp time.Time
}

// 追加到日志
log = append(log, DocOperation{Version: 101, UserID: "user_a", Op: insert("X", 5)})
```

### 2.7 Go 实现库（OT 备选）

| 库 | 地址 | 说明 |
|----|------|------|
| Yjs | github.com/yjs/yjs | 当前推荐的前端 CRDT 协作框架 |
| y-websocket | github.com/yjs/y-websocket | Yjs 官方 WebSocket 同步服务参考实现 |
| go-commit | github.com/sourcegraph/go-commit | 轻量，核心 OT 实现 |
| ot/ot | github.com/operational-transformation/ot | 经典 OT 库 |

```go
// 基本用法
transformed, _ := ot.Transform(insertA, insertB)
// transformed[0] 是转换后的 A
// transformed[1] 是转换后的 B
```

### 2.8 在你的平台里的实现

**文档服务模块：**

```
用户打开文档
  → 后端校验文档/会话权限
  → 前端加入 Yjs 文档房间
  → Yjs 增量更新通过 WebSocket 同步
  → 后端定期/按版本持久化文档快照和更新日志
```

**断线重连：** 重连时从最后一个已知版本起，服务器重放增量操作即可。

### 2.9 OT 与 CRDT 对比（进阶知识）

| 对比 | OT | CRDT |
|------|-----|------|
| 原理 | 服务器统一排序，操作转换 | 去中心化，不需要服务器排序 |
| 服务器依赖 | 强依赖服务器 | 可以无服务器（P2P） |
| 实现复杂度 | 中等 | 较复杂 |
| 代表产品 | 飞书文档早期 | Figma、Notion、腾讯文档 |

飞书/Google Docs 历史上更偏 OT/服务端排序方案，Notion/Figma/Yjs 更偏 CRDT。**当前 WorkPal 更适合优先采用 Yjs/CRDT，OT 作为算法学习和备选设计保留。**

## 3. 开放平台（机器人 + Webhook）

### 3.1 什么是开放平台

开放平台让你的应用可以被第三方开发者扩展。第三方可以通过**机器人（Bot）**和**Webhook** 与你的平台交互，甚至开发自己的小程序/应用。

**在你的平台里能做什么：**
- GitHub 有新 PR → 自动通知到工作群
- 有人@机器人问天气 → 机器人自动回复
- 企业内部系统 → 对接审批流程机器人
- 开发者 → 写代码调用你的 API，发消息给任意群

### 3.2 机器人（Bot）

**机器人**是一个"自动化的用户"，可以：
- 加入群聊，被 @时回复消息
- 接收私信，回复查询结果
- 主动推送消息（定时任务触发）

**工作原理：**
```
用户: @机器人 天气怎么样
        ↓
你的服务器（收到 @消息事件）
        ↓
解析意图 → 调用天气 API
        ↓
发消息到群里：今天 25°C，晴
```

**事件驱动架构：**

```go
// 定义机器人可以处理的事件
type BotEvent struct {
    Type    string    `json:"type"`    // "mention" | "direct_message" | "command"
    UserID  string    `json:"user_id"`
    Content string    `json:"content"` // 用户说的内容
    BotID   string    `json:"bot_id"`
}

// 机器人处理函数
type BotHandler func(event *BotEvent) string

// 注册机器人
func RegisterBot(token string, handler BotHandler) error
```

### 3.3 Webhook

**Webhook** 是一种"反向调用"机制：你的服务器提供一个 HTTP 接口，外部系统在这个接口触发事件时主动调用它。

**应用场景举例：**

| 外部事件 | 触发你的 Webhook | 你做什么 |
|---------|----------------|---------|
| GitHub PR 创建 | POST /webhook/github | 发通知到 #dev 群 |
| 监控系统报警 | POST /webhook/alert | 发通知到 #ops 群 |
| 定时任务（每天9点） | 你的 cron 触发 | 发早报到 #general 群 |
| CRM 客户付款成功 | POST /webhook/payment | 发工单给对应销售 |

**Webhook 安全性：**
外部系统调用你的 Webhook 时，需要验证来源，防止被恶意调用：

```go
// 验证 GitHub Webhook 签名
func VerifyGitHubWebhook(payload []byte, signature, secret string) bool {
    mac := hmac.New(sha256.New, []byte(secret))
    mac.Write(payload)
    expected := "sha256=" + hex.EncodeToString(mac.Sum(nil))
    return hmac.Equal([]byte(expected), []byte(signature))
}

// 中间件验证
router.POST("/webhook/github", verifyWebhook, githubWebhookHandler)
```

### 3.4 OAuth2 第三方授权

如果第三方开发者想开发小程序/应用，需要通过 **OAuth2** 获取用户授权：

```
第三方应用 → 跳转到你的平台授权页
  → 用户点击"授权"
  → 你的平台返回授权码给第三方
  → 第三方用授权码换 Access Token

// 2. 第三方用授权码换 Token

    // 验证 code 有效，删除 code，生成 access_token 返回

```

**整体架构：**

## 4. 监控与 CI（Prometheus + GitHub Actions）

### 4.1 Prometheus 监控

#### 什么是 Prometheus

**Prometheus** 是一个开源的**时序数据库**，专门用来存储"随时间变化的指标数据"，比如：
- 服务器 CPU 使用率（每 10 秒一次）
- HTTP 请求量（每分钟累计计数）
- 函数执行时间（毫秒级）

它的特点是：
- **拉模式（Pull）**：Prometheus 主动从你的服务拉指标，不是你的服务推送
- **只存时序数据**：不适合存日志/文档
- **自带 PromQL 查询语言**：灵活查询
- **生态丰富**：Grafana 专门可视化 Prometheus 数据

#### 在 Go 服务里集成

```go
import (
    "github.com/prometheus/client_golang/prometheus"
    "github.com/prometheus/client_golang/prometheus/promhttp"
    "github.com/prometheus/client_golang/prometheus/promauto"
)

var (
    // HTTP 请求计数
    httpRequestsTotal = promauto.NewCounterVec(
        prometheus.CounterOpts{
            Name: "http_requests_total",
            Help: "Total number of HTTP requests",
        },
        []string{"method", "path", "status"},
    )

    // HTTP 请求延迟
    httpRequestDuration = promauto.NewHistogramVec(
        prometheus.HistogramOpts{
            Name:    "http_request_duration_seconds",
            Help:    "HTTP request latency",
            Buckets: prometheus.DefBuckets,
        },
        []string{"method", "path"},
    )
)

// 中间件自动记录
func PrometheusMiddleware() gin.HandlerFunc {
    return func(c *gin.Context) {
        start := time.Now()
        
        c.Next()
        
        duration := time.Since(start).Seconds()
        httpRequestsTotal.WithLabelValues(
            c.Request.Method,
            c.FullPath(),
            strconv.Itoa(c.Writer.Status()),
        ).Inc()
        httpRequestDuration.WithLabelValues(
            c.Request.Method,
            c.FullPath(),
        ).Observe(duration)
    }
}

func main() {
    router := gin.New()
    // 暴露 /metrics 端点，Prometheus 会定时来拉
    router.GET("/metrics", gin.WrapH(promhttp.Handler()))
    router.Use(PrometheusMiddleware())
}
```

#### 在你的平台里看什么指标

| 指标类型 | 具体指标 | 意义 |
|---------|---------|------|
| **请求量** | `http_requests_total{method, path}` | 哪些接口被访问最频繁 |
| **延迟** | `http_request_duration_seconds{p99}` | 用户抱怨慢时，定位到哪个接口 |
| **错误率** | `http_requests_total{status=5xx}` | 是否有大量报错 |
| **在线人数** | `websocket_connections_active` | 实时在线用户数 |
| **消息量** | `messages_sent_total` | 消息发送速率 |
| **数据库慢查询** | `db_query_duration_seconds` | 数据库性能 |

#### Grafana 可视化

Prometheus 本身没有界面，需要搭配 **Grafana** 一起用：

```
你的 Go 服务:8080/metrics
        ↓ Prometheus 每 15s 拉一次
    Prometheus DB（存储时序数据）
        ↓ 查询
    Grafana（Web 界面，图表/看板）
```

Grafana 有现成的 dashboard 模板（比如 "Go Runtime"），导入即可，不需要自己画图。

### 4.2 GitHub Actions CI

#### 什么是 CI，CD 后续再接入

- **CI（Continuous Integration，持续集成）**：代码提交后自动编译、跑测试，确保代码能构建通过
- **CD（Continuous Deployment，持续部署）**：CI 通过后自动把新版本部署到服务器

**没有 CI 时：**
```
你: 本地改代码 → 手动打包 → 上传到服务器 → 手动重启
    每天重复 N 次，累死了
```

**有了当前 CI 后：**
```
你: git push
    当前自动：编译 → 测试 → 类型检查
    后续可扩展：打包 → 部署
```

#### GitHub Actions 基本概念

| 概念 | 说明 |
|------|------|
| **Workflow（工作流）** | 一个自动化流程，定义在 `.github/workflows/` 目录下 |
| **Job（任务）** | 工作流里的一个步骤，多个 Job 可以并行 |
| **Step（步骤）** | 每个 Job 里的具体动作（执行命令/调用 Action） |
| **Action（动作）** | 可复用的步骤单元，GitHub 社区有大量现成 Action |

#### 你的项目里的 Workflow

```yaml
# .github/workflows/ci.yml

name: CI

on:
  push:
    branches: [main]
  pull_request:
    branches: [main]

jobs:
  backend:
    runs-on: ubuntu-latest
    services:
      postgres:
        image: postgres:16
      redis:
        image: redis:7
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-go@v5
        with:
          go-version: '1.22'
          cache-dependency-path: backend/go.sum
      - name: Download dependencies
        working-directory: ./backend
        run: go mod download && go mod verify
      - name: Build
        working-directory: ./backend
        run: go build -v ./...
      - name: golangci-lint
        uses: golangci/golangci-lint-action@v4
        with:
          working-directory: ./backend
      - name: Test with race detector
        working-directory: ./backend
        run: go test -race -v ./...

  frontend:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with:
          node-version: '20'
          cache: 'npm'
          cache-dependency-path: frontend/package-lock.json
      - name: Install dependencies
        working-directory: ./frontend
        run: npm ci
      - name: Type check
        working-directory: ./frontend
        run: npx tsc --noEmit
      - name: Run tests
        working-directory: ./frontend
        run: npm test
```

当前仓库只做 CI，不自动构建镜像或部署生产环境。Docker 镜像推送、SSH 部署、生产发布审批可以在后续有真实部署目标后再补。

#### 在你的平台里的效果

```
你本地改代码 → git push / PR
  → GitHub Actions 自动运行
      → 后端 go build / golangci-lint / go test -race
      → 前端 npm ci / TypeScript 类型检查 / Vitest
  → 检查通过后再合并或发布
```

#### secrets 怎么配置（后续部署阶段）

当前 CI 不需要部署密钥。如果后续要推送镜像并部署服务器，可在 GitHub 仓库 → Settings → Secrets and variables → Actions 里配置：

| Secret 名 | 用途 |
|----------|------|
| `DOCKER_USERNAME` | 可选，发布 Docker 镜像时使用 |
| `DOCKER_TOKEN` | 可选，发布 Docker 镜像时使用 |
| `SERVER_HOST` | 可选，部署到服务器时使用 |
| `SERVER_USER` | 可选，部署到服务器时使用 |
| `SERVER_SSH_KEY` | 可选，部署到服务器时使用 |

## 5. 阶段四学习路线与优先级

### 推荐顺序

```
已完成: Prometheus /metrics + GitHub Actions CI
         ↓ 当前先保障可观测和代码质量
下一步: 机器人 + Webhook
         ↓ 对外开放接口，第三方可用
再后续: 云文档协作（Yjs/CRDT，OT 作为对比学习）
         ↓ 音视频（WebRTC）← 需要 TURN 服务器，有难度
```

### 为什么是这个顺序

| 顺序 | 原因 |
|------|------|
| 1. Prometheus | 已落地 `/metrics`，后续可补 Grafana 看板 |
| 2. GitHub Actions | 已落地 CI，后续可补 Docker 镜像和部署 |
| 3. 机器人 + Webhook | 外部价值高，让平台真正可以被别人集成 |
| 4. Yjs/CRDT + WebRTC | 难度最高，需要前面积累的实时同步和权限模型 |

## 6. 快速参考

### 各技术核心概念速记

| 技术 | 核心概念 | 一句话描述 |
|------|---------|-----------|
| WebRTC | STUN/TURN/ICE | 帮两个 NAT 后的电脑找到彼此 |
| Yjs/CRDT | 增量同步、冲突合并 | 当前推荐的云文档协作方向 |
| OT 算法 | Transform | 协作编辑原理学习和备选方案 |
| Webhook | 反向 HTTP 调用 | 外部事件主动调用你的 HTTP 接口 |
| 机器人 | 事件驱动 | 有人发消息 → 你的服务器处理 → 回复 |
| Prometheus | 拉模式时序数据库 | Prometheus 主动来拉你的指标 |
| GitHub Actions | 自动化流水线 | push/PR 代码 → 自动测试和构建，部署后续接入 |
