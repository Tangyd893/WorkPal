# WorkPal 测试手册

这份手册用于验收当前项目是否已经按代码真实情况跑通。

## 1. 环境要求

- Go `1.22+`
- Node.js `18+`
- npm
- Docker Desktop 或 Docker Engine

先确认 Docker 正在运行：

```powershell
docker version
```

只有输出同时包含 `Client` 和 `Server` 时，再继续下面步骤。

## 3. 启动方式

### 3.1 整套容器启动

在仓库根目录执行：

```powershell
docker compose -f docker/docker-compose.yaml build
docker compose -f docker/docker-compose.yaml up -d
docker compose -f docker/docker-compose.yaml ps
```

预期：

- `postgres`、`redis`、`minio` 为 `Up` 或 `healthy`
- `gateway`、`user-service`、`im-service`、`file-service`、`search-service`、`workspace-service` 为 `Up`

当前 Compose 会让开启注册发现的后端服务等待 Redis 健康后再启动，这样网关启动后更容易直接看到 Redis 注册表中的服务实例。

## 5. 网关与服务发现验收

执行：

```powershell
Invoke-RestMethod http://localhost:8080/health/live
Invoke-RestMethod http://localhost:8080/health/ready
Invoke-RestMethod http://localhost:8080/health
Invoke-RestMethod http://localhost:8080/gateway/routes
Invoke-RestMethod http://localhost:8080/gateway/services
```

重点检查：

1. `live` 是否返回网关存活
2. `ready` 是否返回下游就绪
3. `routes` 是否返回显式路由目录
4. `services` 是否返回 `discovery_mode`、实例信息、超时、重试和熔断元数据

使用 `admin / admin123` 登录前端后，按下面顺序检查：

2. 总览中的卡片和区块可以跳转到对应模块
3. 沟通模块可以查看会话列表
4. 任务模块可以新增、推进、重置、删除、分享
5. 日程模块可以新增、删除、分享
6. 文件模块可以上传、分享、删除
7. 通讯录模块可以按部门筛选，并按姓名、电话、职称、工号、部门搜索
8. 设置面板可以切换语言、主题、提示音、密度

## 9. IM 实时与搜索链路验收

当前 IM 链路至少应满足以下预期：

1. 消息发送成功后，前端实时会话能收到消息
2. 多个 IM 实例之间通过 Redis Pub/Sub 扇出实时消息
3. IM 会把消息事件先写入 `message_outbox`
4. 后台 worker 再把 outbox 事件投递到 Redis Streams
5. Search Service 消费 Redis Streams 后更新 Bleve 索引

这条链路的意义是：搜索索引更新不再依赖请求线程里“顺手发事件”。


## 11. 验收时最该关注什么

不是只看“项目能不能跑”，还要看：

1. Gateway 是否真的暴露了服务目录
2. 服务是否真的注册到了 Redis
3. 文件模块是否已经只使用后端真实数据
4. IM 实时层是否已经脱离纯单实例内存态
5. 搜索更新链路是否已经升级为 outbox + Redis Streams

## 12. 前端非功能检查点

当前前端非功能基线包括：

1. 包管理统一使用 npm 和 `package-lock.json`
2. `npm run lint` 能作为静态质量门禁执行
3. 登录页、设置抽屉、文件面板有组件测试覆盖关键交互和语义属性
4. 错误提示使用 `role="alert"`，加载和普通提示使用 `role="status"`
5. 设置抽屉和创建会话弹窗使用 dialog 语义
6. 分段按钮、会话选择、导航选中态有明确的可访问状态
7. 样式层支持键盘 `focus-visible` 和 `prefers-reduced-motion`
