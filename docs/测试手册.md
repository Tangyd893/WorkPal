# WorkPal 测试手册

本文档只描述当前仓库代码已经具备的能力，以及按代码实际情况验证项目的方法。

## 1. 测试目标

本轮项目的验收重点有三类：

1. 后端已经是“API Gateway + 领域微服务”的唯一运行形态
2. Gateway 具备基础治理能力与管理面，而不只是转发入口
3. 前端工作台能通过真实后端接口完成核心协作操作

## 2. 环境要求

- Go `1.22+`
- Node.js `18+`
- npm
- Docker Desktop 或 Docker Engine

开始前先确认 Docker 真的在运行：

```powershell
docker version
```

只有输出同时包含 `Client` 和 `Server` 时，才继续后续步骤。

## 4. 启动方式

### 4.1 推荐方式：整套 Docker Compose

在仓库根目录执行：

```powershell
docker compose -f docker/docker-compose.yaml build
docker compose -f docker/docker-compose.yaml up -d
docker compose -f docker/docker-compose.yaml ps
```

预期结果：

- `postgres`、`redis`、`minio` 为 `Up` 或 `healthy`
- `gateway`、`user-service`、`im-service`、`file-service`、`search-service`、`workspace-service` 为 `Up`

首次启动时，以下服务会自动创建自己的数据库：

- `workpal_user`
- `workpal_im`
- `workpal_file`
- `workpal_workspace`

### 4.2 学习方式：基础设施用 Docker，服务从源码运行

如果你想逐个调试服务，可以先只启动基础设施：

```powershell
docker compose -f docker/docker-compose.yaml up -d postgres redis minio
```

然后分别打开多个终端，在 `backend` 目录启动：

```powershell
go run ./cmd/user-service
```

```powershell
go run ./cmd/im-service
```

```powershell
go run ./cmd/file-service
```

```powershell
go run ./cmd/search-service
```

```powershell
go run ./cmd/workspace-service
```

```powershell
go run ./cmd/gateway
```

最后在 `frontend` 目录启动前端：

```powershell
npm ci
npm run dev -- --host 127.0.0.1
```

## 7. 启动后快速自检

### 7.1 Gateway 健康检查

```powershell
Invoke-RestMethod http://localhost:8080/health/live
Invoke-RestMethod http://localhost:8080/health/ready
Invoke-RestMethod http://localhost:8080/health
```

预期：

- `/health/live` 返回 `status=ok`
- `/health/ready` 返回 `status=ok`
- `/health` 返回 `status=ok`

如果任一微服务不可用，`/health/ready` 和 `/health` 会返回 `degraded`，这是当前代码的真实行为。

### 7.2 Gateway 管理面

```powershell
Invoke-RestMethod http://localhost:8080/gateway/routes
Invoke-RestMethod http://localhost:8080/gateway/services
```

预期：

- `/gateway/routes` 返回显式路由目录
- `/gateway/services` 返回 5 个下游服务
- 服务目录中能看到超时、重试次数、熔断器状态

### 7.3 登录接口

```powershell
$body = @{
  username = "admin"
  password = "admin123"
} | ConvertTo-Json

Invoke-RestMethod `
  -Uri "http://localhost:8080/api/v1/auth/login" `
  -Method Post `
  -ContentType "application/json" `
  -Body $body
```

预期：

- `code` 为 `0`
- `data.token` 存在

### 7.4 前端入口

打开：

```text
http://localhost:3000
```

使用 `admin / admin123` 登录，预期会跳转到：

```text
http://localhost:3000/workspace/overview
```

## 10. 常见问题

### 10.1 `/health/ready` 或 `/health` 返回 `degraded`

说明不是 Gateway 自己坏了，而是某个下游服务不可用。直接查看返回中的 `components` 和 `errors` 字段能定位具体服务。

### 10.2 `/gateway/services` 正常，但业务接口报错

优先检查：

1. `/gateway/routes` 中该路由是否存在
2. 返回头里是否有 `X-Upstream-Service`
3. 目标微服务日志是否有对应请求

### 10.3 第一次启动较慢

这是正常现象。首次启动会涉及：

- Go 镜像构建
- PostgreSQL、Redis、MinIO 初始化
- 各服务自动创建自己的数据库
- User Service 自动补齐部门、员工、验收账号
