# WorkPal 测试手册

本文档只描述当前仓库代码已经具备的能力，以及按代码实际情况验证项目的方法。

## 1. 测试目标

本轮代码的验收重点有两类：

1. 后端已经是“API Gateway + 领域微服务”的唯一运行形态。
2. 前端的协作工作台能够基于真实后端接口完成登录、沟通、任务、日程、文件、通讯录等核心操作。

## 2. 环境要求

- Go `1.22+`
- Node.js `18+`
- npm
- Docker Desktop 或 Docker Engine

开始前先确认 Docker 真的在运行：

```powershell
docker version
```

只有输出同时包含 `Client` 和 `Server` 时，才继续后续步骤。

## 4. 启动方式

### 4.1 推荐方式：整套 Docker Compose

在仓库根目录执行：

```powershell
docker compose -f docker/docker-compose.yaml build
docker compose -f docker/docker-compose.yaml up -d
docker compose -f docker/docker-compose.yaml ps
```

预期结果：

- `postgres`、`redis`、`minio` 为 `Up` 或 `healthy`
- `gateway`、`user-service`、`im-service`、`file-service`、`search-service`、`workspace-service` 为 `Up`

说明：

- 首次启动时，`user-service`、`im-service`、`file-service`、`workspace-service` 会自动创建自己负责的数据库。
- 自动创建的数据库名分别是：
  - `workpal_user`
  - `workpal_im`
  - `workpal_file`
  - `workpal_workspace`

### 4.2 学习方式：基础设施用 Docker，服务从源码运行

如果你想逐个调试服务，可以先只启动基础设施：

```powershell
docker compose -f docker/docker-compose.yaml up -d postgres redis minio
```

然后分别打开多个终端，在 `backend` 目录启动服务：

```powershell
go run ./cmd/user-service
```

```powershell
go run ./cmd/im-service
```

```powershell
go run ./cmd/file-service
```

```powershell
go run ./cmd/search-service
```

```powershell
go run ./cmd/workspace-service
```

```powershell
go run ./cmd/gateway
```

最后在 `frontend` 目录启动前端：

```powershell
npm ci
npm run dev -- --host 127.0.0.1
```

## 5. 配置与数据归属

当前后端已经不再提供一体化单体入口，服务边界如下：

| 服务 | 持久化归属 | 关键职责 |
| --- | --- | --- |
| Gateway | 无状态 | 统一入口、反向代理、限流、请求 ID、聚合健康检查 |
| User Service | `workpal_user` | 登录、用户、部门、员工档案、开发种子数据 |
| IM Service | `workpal_im` | 私聊、群聊、消息、群公告、WebSocket |
| File Service | `workpal_file` | 个人文件、群文件、分享链接 |
| Search Service | Bleve + Redis Streams | 消息索引、搜索 |
| Workspace Service | `workpal_workspace` | 任务、日程 |

## 7. 启动后快速自检

### 7.1 网关健康检查

```powershell
Invoke-RestMethod http://localhost:8080/health
```

预期：

- HTTP 状态码为 `200`
- `status` 为 `ok`
- `components` 中能看到五个下游服务

如果任一微服务不可用，网关 `/health` 会返回 `degraded`，这是当前代码的真实行为。

### 7.2 登录接口

```powershell
$body = @{
  username = "admin"
  password = "admin123"
} | ConvertTo-Json

Invoke-RestMethod `
  -Uri "http://localhost:8080/api/v1/auth/login" `
  -Method Post `
  -ContentType "application/json" `
  -Body $body
```

预期：

- `code` 为 `0`
- `data.token` 存在

### 7.3 前端入口

打开：

```text
http://localhost:3000
```

使用 `admin / admin123` 登录，预期会跳转到：

```text
http://localhost:3000/workspace/overview
```

## 10. 常见问题

### 10.1 `/health` 返回 `degraded`

说明不是网关自己坏了，而是某个下游服务不可用。直接查看 `components` 和 `errors` 字段能定位到具体服务。

### 10.2 能访问前端但接口报错

优先检查：

1. `http://localhost:8080/health`
2. `docker compose -f docker/docker-compose.yaml ps`
3. 网关和对应微服务是否都已启动

### 10.3 第一次启动较慢

这是正常现象。首次启动会涉及：

- Go 镜像构建
- PostgreSQL、Redis、MinIO 初始化
- 各服务自动创建自己的数据库
- User Service 自动补齐部门、员工、验收账号
