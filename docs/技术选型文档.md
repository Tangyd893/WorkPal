# WorkPal 技术选型文档

本文档只讨论当前代码已经采用的技术方案，以及为什么这样选。

## 1. 设计目标

WorkPal 不是单纯的聊天 Demo，而是一个适合学习“办公协作平台工程化实现”的项目。技术选型围绕四个目标展开：

1. 本地可完整跑通
2. 服务边界清晰，便于学习微服务拆分
3. 不把基础设施复杂度拉到过高
4. 保留继续演进到生产级方案的路径

## 2. 总体技术栈

### 2.1 后端

- 语言：Go
- Web 框架：Gin
- ORM：GORM
- 关系型数据库：PostgreSQL
- 缓存与事件总线：Redis + Redis Streams
- 搜索：Bleve
- 对象存储：MinIO，带本地文件存储回退

### 2.2 前端

- React 18
- Vite
- TypeScript
- Zustand
- Axios
- Playwright

## 3. 为什么后端用 Go

### 3.1 适合服务化拆分

Go 语言在以下方面很适合这个项目：

- 编译后部署简单
- 并发模型天然适合 WebSocket、消息消费、网关代理
- 标准库和生态足够支撑中小型微服务系统

### 3.2 适合学习服务边界

这个项目的服务数量已经足够体现真实问题：

- 认证和目录
- 即时通信
- 文件
- 搜索
- 任务与日程
- API Gateway

用 Go 写这些服务，能比较直接地看到：

- 每个进程如何独立启动
- 每个服务如何独立持有数据
- 同步调用与异步事件如何搭配

## 4. 为什么用 Gin 而不是更重的框架

选 Gin 的原因：

- 足够轻
- 路由、中间件、JSON 输出直接
- 学习成本低
- 与 Go 常见工程结构兼容

这个项目需要的能力主要是：

- REST API
- 中间件
- WebSocket 接入点
- 健康检查和指标暴露

Gin 正好覆盖这些核心需求，没有引入额外框架心智负担。

## 5. 为什么用 GORM

GORM 的定位不是“最极致性能”，而是“足够高效的业务开发工具”。在当前项目里它的价值主要在：

- 模型定义和迁移简单
- 仓储层写法清晰
- 降低样板代码
- 适合教学型项目逐步展开

当前代码里，GORM 最适合学习的点不是高级特性，而是：

- 模型映射
- 仓储查询封装
- 迁移责任按服务拆分

## 6. 为什么选 PostgreSQL

PostgreSQL 在这个项目里承担“结构化业务事实”的角色。

选择它的原因：

- 对 JSON、索引、事务支持成熟
- 本地开发体验好
- 对办公协作类数据结构很友好

更重要的是，这个项目现在已经把数据库归属按服务拆开：

| 服务 | 数据库 |
| --- | --- |
| User Service | `workpal_user` |
| IM Service | `workpal_im` |
| File Service | `workpal_file` |
| Workspace Service | `workpal_workspace` |

这样做的收益是：

- 服务的数据边界更清楚
- 学习时能区分“服务拆分”和“数据库拆分”是两件相关但不同的事
- 以后想替换某个服务的存储实现，冲击面更小

## 7. 为什么选 Redis，而不是一上来上 Kafka

### 7.1 当前阶段的目标

这个项目现在需要的消息能力主要是：

- IM Service 发布消息索引事件
- Search Service 订阅并消费这些事件
- 本地开发能快速启动

Redis Streams 在这个阶段有几个优势：

- 部署轻
- 学习门槛比 Kafka 更低
- 足够体现异步解耦、消费组、重试、死信这些核心概念

### 7.2 当前承担的职责

Redis 在当前代码里承担两类职责：

1. 缓存基础能力
2. Redis Streams 事件总线

### 7.3 为什么不是 Kafka

Kafka 更适合更大吞吐量、更复杂的消费拓扑，但对于这个项目：

- 运维复杂度明显上升
- 本地启动成本更高
- 会把学习重心从“协作平台业务”拉到“消息基础设施”上

因此当前选 Redis Streams 更平衡。

## 8. 为什么搜索用 Bleve

当前消息搜索的目标是：

- 在本地即可运行
- 能体现“写入主库”和“异步建索引”的解耦
- 不强依赖额外搜索集群

Bleve 的优势：

- 纯 Go
- 嵌入式
- 本地学习成本低

因此 Search Service 的角色是：

- 从 Redis Streams 消费消息事件
- 更新 Bleve 索引
- 提供搜索 API

如果以后要演进到更强方案，可以替换为 Elasticsearch 或 OpenSearch，但当前没有必要。

## 9. 为什么文件服务用 MinIO + 本地回退

文件服务需要体现“元数据”和“文件内容”分离：

- PostgreSQL 存元数据
- MinIO 存文件对象

这样能学习到对象存储的基本模式，同时又保留本地回退：

- MinIO 初始化失败时，仍可退回本地目录存储
- 降低本地调试门槛

这很适合教学项目，因为它同时展示了：

- 抽象接口
- 存储实现切换
- 元数据与二进制内容分离

## 10. 为什么前端选 React + Vite + Zustand

### 10.1 React

适合模块化工作台界面。当前前端不是单页静态页面，而是：

- 总览
- 沟通
- 任务
- 日程
- 文件
- 通讯录

React 很适合把这些板块拆成独立面板和可复用组件。

### 10.2 Vite

Vite 的价值主要是本地开发体验：

- 启动快
- 热更新快
- 配置简单

### 10.3 Zustand

这个项目的前端状态不需要 Redux 那样更重的机制。Zustand 足以处理：

- 鉴权状态
- 偏好设置
- 聊天控制器

这样既保留了状态管理概念，又不会让学习曲线过陡。

## 11. 为什么通过 API Gateway 对前端暴露统一入口

当前前端始终只访问：

```text
http://localhost:8080
```

这样做有三个好处：

1. 前端代理配置简单
2. 服务拆分后前端仍然稳定
3. 更接近真实生产入口模式

Gateway 当前负责：

- 路由分发
- 反向代理
- 请求 ID
- 限流
- 聚合健康检查

这让它不是“转发器摆设”，而是一个真正有职责的边缘服务。

## 12. 当前技术组合的优点

### 12.1 学习路径清晰

你可以从最表层一路看到：

- UI 模块
- API 封装
- 网关分发
- 领域服务
- 数据库
- 异步索引

### 12.2 本地可运行

对学习项目来说，“能跑通”比“技术最潮”更重要。当前这套组合可以在一台开发机完整启动。

### 12.3 有演进空间

当前技术栈并不是终点，但它保留了明确的替换路径：

- Redis Streams -> Kafka
- Bleve -> Elasticsearch / OpenSearch
- 单机 WebSocket Hub -> 分布式连接层
- MinIO 本地开发 -> 云对象存储

## 13. 当前取舍与限制

### 13.1 服务仍共享一套基础设施实例

现在是“独立数据库 + 共享 PostgreSQL/Redis/MinIO 实例”，而不是每个服务独占一套中间件。这是为了平衡本地开发成本。

### 13.2 搜索不是生产级全文检索方案

Bleve 更偏学习和轻量部署，不适合作为高规模生产搜索的最终方案。

### 13.3 WebSocket 仍是单实例内存 Hub

IM Service 已经独立，但连接管理还没有做水平扩展。

## 14. 结论

当前这套技术选型的核心价值，在于它已经足够像一个真实的办公协作平台后端，但仍然保持了本地可学、可跑、可拆解的节奏。

如果你的目标是学习“业务系统如何从模块化单体走向真正的微服务项目”，这套选型是合适的。
