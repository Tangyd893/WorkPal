# WorkPal 技术选型文档

本文档只讨论当前代码已经采用的技术方案，以及这些方案为什么适合这个项目。

## 1. 技术目标

WorkPal 的目标不是堆砌“最重”的微服务基础设施，而是在本地可跑通的前提下，把协作平台最核心的工程问题做出来：

1. 统一入口
2. 领域拆分
3. 服务边界
4. 异步解耦
5. 基础治理

## 2. 总体技术栈

### 2.1 后端

- Go
- Gin
- GORM
- PostgreSQL
- Redis
- Redis Streams
- Bleve
- MinIO

### 2.2 前端

- React 18
- Vite
- TypeScript
- Zustand
- Axios
- Playwright

## 3. 为什么后端用 Go

Go 很适合这个项目的原因有三点：

1. 编译部署简单
2. 并发模型适合 WebSocket、消息消费、网关代理
3. 能把微服务的结构问题展示得比较直接

对学习者来说，这意味着你会更容易把注意力放在：

- 服务边界
- 同步调用
- 异步事件
- 入口治理

而不是先陷入过重的平台框架心智里。

## 4. 为什么用 Gin

这个项目需要的核心后端能力主要是：

- REST API
- 中间件
- WebSocket 接入
- 健康检查
- 指标暴露

Gin 的特点是：

- 足够轻
- 路由和中间件组织直接
- 和当前分层结构兼容

它很适合作为教学型微服务项目的 Web 框架。

## 5. 为什么用 GORM

GORM 在这个项目里承担的是“业务开发效率工具”的角色，而不是极致性能方案。

当前它最适合学习的点是：

- 模型映射
- 仓储封装
- 服务负责自己的迁移

这三点对理解微服务中的数据归属，比讨论 ORM 高级特性更重要。

## 6. 为什么选 PostgreSQL

PostgreSQL 在当前架构里的角色是“结构化业务事实存储”。

关键原因：

- 事务能力成熟
- 索引和 JSON 支持足够
- 本地开发体验好

更重要的是，这个项目现在已经把数据库按服务拆开：

| 服务 | 数据库 |
| --- | --- |
| User Service | `workpal_user` |
| IM Service | `workpal_im` |
| File Service | `workpal_file` |
| Workspace Service | `workpal_workspace` |

这比“多个服务共享一个应用数据库”更适合学习微服务边界。

## 7. 为什么选 Redis + Redis Streams

Redis 在当前系统里承担两类职责：

1. 缓存基础能力
2. 消息事件总线

Redis Streams 被选中的原因是：

- 部署轻
- 本地学习成本低
- 足够体现消费组、重试、死信、异步解耦这些关键概念

在当前代码里，它主要承担：

- IM Service 发布消息索引事件
- Search Service 订阅并消费这些事件

## 8. 为什么搜索用 Bleve

当前消息搜索目标是：

- 本地就能跑
- 能体现索引与主业务写入解耦
- 不依赖额外搜索集群

Bleve 的优势：

- 纯 Go
- 嵌入式
- 适合教学和轻量部署

所以 Search Service 当前的价值，是让你看见：

- 为什么搜索不一定和主业务写库绑死
- 为什么事件驱动索引更新是合理的

## 9. 为什么文件存储用 MinIO + 本地回退

文件服务需要展示一个很重要的设计点：

- 文件元数据和文件内容分离

当前做法是：

- PostgreSQL 存元数据
- MinIO 存对象内容
- MinIO 不可用时回退到本地文件目录

这让项目既能展示对象存储抽象，又不会把本地调试门槛拉高。

## 10. 为什么前端选 React + Vite + Zustand

### 10.1 React

因为前端不是单页静态展示，而是一个工作台壳，里面要承载：

- 总览
- 沟通
- 任务
- 日程
- 文件
- 通讯录

React 很适合这种模块化工作台。

### 10.2 Vite

Vite 的价值主要在开发体验：

- 启动快
- 热更新快
- 配置负担小

### 10.3 Zustand

当前前端状态不需要更重的 Redux 体系，Zustand 足够处理：

- 鉴权状态
- 偏好设置
- 聊天控制器状态

## 11. 为什么后端必须有 API Gateway

前端当前只访问一个后端地址：

```text
http://localhost:8080
```

Gateway 的存在不是为了“看起来像微服务”，而是为了把入口层职责单独拿出来。

当前 Gateway 已经负责：

- 路由目录
- 下游服务目录
- 请求 ID
- 限流
- 服务级超时
- 只对幂等读请求启用重试
- 熔断器
- 存活 / 就绪 / 聚合健康检查
- 路由管理面：`/gateway/routes`
- 服务管理面：`/gateway/services`

这使它同时承担了两类角色：

1. Spring Cloud Gateway 式的入口层
2. Sentinel 式的基础流量治理入口

## 12. 用 Spring Cloud Alibaba 的视角看当前选型

如果你熟悉 Java 微服务，可以这样映射：

| Spring Cloud Alibaba 视角 | 当前 Go 实现 |
| --- | --- |
| Spring Cloud Gateway | `backend/cmd/gateway` |
| Nacos（静态配置驱动版本） | `backend/configs/*` + `/gateway/services` |
| Sentinel | Gateway 中的限流、重试、熔断 |
| OpenFeign / 服务间调用 | `backend/internal/clients/*` |
| RocketMQ | Redis Streams |

这份映射的意义在于：

- 你可以用成熟的微服务认知框架来理解当前 Go 代码
- 也能看见“没有完整 Java 全家桶时，Go 项目怎么把核心思想落地”

## 13. 当前技术组合的优点

### 13.1 学习路径清晰

你可以顺着一条链路看到：

- 前端页面
- API 封装
- Gateway
- 领域服务
- 数据库
- 异步搜索

### 13.2 微服务味道已经真实存在

当前系统已经具备：

- 统一入口
- 领域服务拆分
- 服务自有数据库
- 异步消息解耦
- 基础流量治理

### 13.3 本地仍然容易启动

对学习项目来说，“能跑通”很重要。当前这套组合不要求你先搭一整套重型平台。

## 14. 当前保留的取舍

### 14.1 还没有真正的动态注册中心

现在的服务目录是静态配置驱动，而不是动态服务注册。

### 14.2 Gateway 还不是生产级完整入口平台

虽然已经有治理能力，但还没有继续扩展到：

- 动态配置热更新
- 更细粒度的路由级策略
- 更丰富的追踪与观测

### 14.3 WebSocket 仍是单实例连接层

IM Service 的 WebSocket Hub 还没有进化到跨实例广播。

## 15. 结论

当前 WorkPal 的技术选型已经足够构成一套“轻量但真实”的微服务学习样本。

它最有价值的地方，不是把所有平台能力都一次做满，而是：

- 把入口层做出来了
- 把服务边界做清楚了
- 把消息解耦做出来了
- 把治理能力放到了可读、可测、可讲解的范围内

这正适合拿来系统学习微服务。
