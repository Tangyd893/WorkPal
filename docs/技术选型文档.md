# WorkPal 技术选型文档

## 1. 选型目标

本项目的技术选型优先满足三件事：

1. 适合学习微服务核心概念
2. 能在本地稳定跑通
3. 复杂度不要高到遮住代码本身

换句话说，WorkPal 追求的是“足够真实，又能读懂”，而不是一次性堆满全部企业级基础设施。

## 2. 后端语言与框架

### Go

选择 Go 的原因：

- 并发模型适合网关、WebSocket、消息投递
- 二进制部署简单
- 适合用较少框架展示微服务核心职责

### Gin

选择 Gin 的原因：

- 路由和中间件模型清晰
- 足够轻量
- 很适合把网关与服务入口层代码直接暴露给学习者阅读

### GORM

选择 GORM 的原因：

- 能快速完成服务级模型落地
- 在教学型项目里，比大量手写 SQL 更容易先建立整体结构
- 仍然保留向显式仓储和事务抽象演进的空间

## 3. 数据存储

### PostgreSQL

选择 PostgreSQL 的原因：

- 适合协作平台这类关系型数据
- 功能完整
- 便于服务按库拆分

当前数据库边界如下：

- `workpal_user`
- `workpal_im`
- `workpal_file`
- `workpal_workspace`

### Bleve

选择 Bleve 的原因：

- 本地可运行，不依赖外部搜索集群
- 足够演示“主业务数据 + 派生搜索索引”的分层思路

### MinIO

选择 MinIO 的原因：

- 对象存储接口清晰
- 本地调试成本低
- 适合演示“文件元数据在数据库，文件内容在对象存储”的结构

## 4. Redis 的四重角色

Redis 是当前项目最值得注意的基础设施选型之一，因为它不只是缓存。

### 4.1 服务注册发现

服务启动后会把实例信息注册到 Redis，Gateway 再从 Redis 读取实例目录。

为什么这样选：

- 能清楚演示“服务注册 / 服务发现”的职责
- 实现成本比直接引入完整注册中心低
- 仍然保留静态配置回退，方便本地学习

### 4.2 IM 跨实例扇出

IM Service 使用 Redis Pub/Sub 做 WebSocket 事件跨实例传播。

为什么这样选：

- 把实时系统从“单实例内存 Hub”提升到“多实例可广播”
- 对学习者来说，比先引入更重的消息中间件更容易读懂

### 4.3 Redis Streams 异步链路

IM 到 Search 通过 Redis Streams 传播消息事件。

为什么这样选：

- 能展示异步派生链路
- 支持消费者组、重试和 dead letter 思路

### 4.4 IM Outbox 的落地点

IM Service 在写消息、编辑消息、撤回消息时，会把待投递事件先写到 `message_outbox`。
后台 worker 再把这些事件发布到 Redis Streams。

为什么这样选：

- 让“业务写库”和“待投递事件入库”落在同一事务里
- 降低 Redis 抖动时消息索引事件丢失的风险
- 更适合学习最终一致性和补偿发布

## 5. Gateway 选型

Gateway 没有直接接入独立网关产品，而是由 Go 自己实现。

原因：

- 便于直接学习网关职责
- 可以在代码里清晰看到路由、限流、超时、重试、熔断各自的位置
- 更适合把 Spring Cloud Gateway / Sentinel 的概念映射到当前实现

当前 Gateway 具备：

- 路由目录
- 服务目录
- Redis 服务发现 + 静态回退
- 限流
- 超时
- 重试
- 熔断
- 健康检查

## 6. 前端选型

### React 18 + Vite + TypeScript

选择原因：

- 属于现代前端项目的常见组合
- 构建和调试速度快
- 适合工作台式多模块界面

### Zustand

选择原因：

- 状态模型轻
- 对鉴权、偏好设置、聊天状态这类局部状态已经足够
- 比引入更重的状态框架更利于学习

## 7. 为什么没有直接选更重的方案

项目没有直接引入 Nacos、Sentinel、RocketMQ，不是因为这些组件不重要，而是因为当前阶段更重视：

- 先把职责学明白
- 让代码可读
- 让本地运行成本可控

当前项目已经用轻量方式表达了这些职责：

- Nacos 风格职责 -> Redis 服务注册发现
- Sentinel 风格职责 -> Gateway 治理
- RocketMQ 风格职责 -> Outbox + Redis Streams 异步链路

## 8. 当前选型的优势

- 足够接近真实微服务
- 仍然适合本地学习和调试
- Redis 承担多类基础设施职责，学习价值高
- 入口治理、实时链路、异步链路都已经成型

## 9. 当前选型的不足

- Redis 同时承担注册发现、实时扇出、Streams，职责较集中
- 还没有独立配置中心
- 还没有链路追踪
- 还没有正式数据库迁移体系

这些不是选型错误，而是项目继续演进的自然方向。
