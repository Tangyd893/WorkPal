# WorkPal 技术选型文档

> 本文档针对 4 核 8 GB 虚拟机环境，定制轻量化技术栈，确保全程开发调试流畅。所有选型均保持与生产级架构相同的编程模型，差别仅在部署形态。

> 当前项目状态：后端已采用 Go 1.22 + Gin + GORM + PostgreSQL 16 + Redis 7/Redis Streams + Bleve + MinIO/本地文件双模式；前端采用 React 18 + TypeScript + Vite 5 + npm。本文档中的“阶段”用于说明演进路线，已完成阶段以当前实现为准。

---

## 1. 环境约束

| 资源 | 规格 | 说明 |
|------|------|------|
| CPU | 4 核 AMD Ryzen 7 5800U | 够用，Go 编译速度极快 |
| 内存 | 7.7 GB（可用约 4.5 GB） | 紧张，不宜运行多个重量级中间件 |
| 磁盘 | 148 GB SSD，剩余 96 GB | 充裕 |
| OS | Ubuntu 24.04 / Linux 6.17 |

**设计原则：**
- 生产级中间件优先选**嵌入进程**或**轻量替代**，避免额外的内存开销
- 所有替代方案保持与生产级架构相同的接口契约，方便后续替换

## 2. 技术选型对比与决策

### 2.1 消息队列

| 方案 | 生产级代表 | 内存占用 | 推荐度 | 说明 |
|------|-----------|---------|--------|------|
| Apache Kafka | LinkedIn | ~2 GB+ | ⭐⭐ | 功能强大，但内存开销大，本环境不选 |
| NATS | NATS.io | ~20 MB | ⭐⭐⭐⭐ | 轻量、高性能、纯 Go，支持 JetStream |
| Redis Streams | Redis | ~100 MB | ⭐⭐⭐⭐ | 复用已有 Redis，无需新启进程 |
| 内存队列（sync.Map） | - | 0 | ❌ | 仅限单机，不能跨进程，测试用 |

**当前决策：使用 Redis Streams。** 代码保留 `pkg/msgqueue` 抽象，后续如需 Kafka/NATS 可替换实现，但当前仓库没有 Kafka/NATS 依赖和部署。

```go
// 接口抽象（方便后续切换 Kafka / NATS）
type MessageQueue interface {
    Publish(ctx context.Context, topic string, msg []byte) error
    Subscribe(topic string, handler func([]byte)) error
}
```

### 2.2 全文搜索引擎

| 方案 | 内存占用 | 推荐度 | 说明 |
|------|---------|--------|------|
| Elasticsearch | ~2 GB+ | ⭐⭐ | 功能全，本环境内存压力大，不选 |
| Meilisearch | ~300 MB | ⭐⭐⭐ | 轻量易用，REST API 友好 |
| Bleve | ~50 MB（嵌入） | ⭐⭐⭐⭐ | 纯 Go 嵌入进程，无需独立部署 |
| SQLite FTS5 | ~10 MB（嵌入） | ⭐⭐⭐ | 零依赖，够用，简单场景首选 |

**当前决策：Bleve。** 当前搜索服务为进程内 Bleve 索引，适合学习项目和轻量部署；后续如消息量增大，再考虑 Elasticsearch/Meilisearch。

```go
// 示例：Bleve 索引创建/打开
index, err := bleve.New(indexPath, bleve.NewIndexMapping())
if err != nil {
    index, err = bleve.Open(indexPath)
}
```

### 2.3 对象存储（文件服务）

| 方案 | 内存占用 | 推荐度 | 说明 |
|------|---------|--------|------|
| MinIO | ~500 MB | ⭐⭐⭐ | S3 兼容，功能完整，需独立进程 |
| 阿里云 OSS | 0 | ⭐⭐⭐⭐ | 云端托管，零运维，适合生产 |
| 本地文件系统 | 0 | ⭐⭐⭐⭐⭐ | 开发阶段首选，无需额外进程 |

**当前决策：MinIO + 本地文件系统双模式。** `file.storeType` 可选择 `minio` 或 `local`；MinIO 初始化失败时会回退到本地存储。

```go
// 文件存储接口抽象
type FileStore interface {
    Upload(ctx context.Context, key string, r io.Reader, size int64, contentType string) error
    Download(ctx context.Context, key string) (io.ReadCloser, error)
    Delete(ctx context.Context, key string) error
    GetURL(ctx context.Context, key string) (string, error)
}

// 本地实现
type LocalFileStore struct {
    BasePath string
}

// MinIO 实现（生产级）
type MinIOFileStore struct {
    bucket string
    client *minio.Client
}
```

### 2.4 WebSocket 长连接

| 方案 | 说明 | 推荐度 |
|------|------|--------|
| gorilla/websocket | 最成熟，广泛验证 | ⭐⭐⭐⭐⭐ |
| melean/rwebsocket | 简洁，无外部依赖 | ⭐⭐⭐ |
| 标准库 net/http | 实验性 ws 支持 | ⭐⭐ |

**决策： gorilla/websocket，无悬念。**

---

### 2.5 数据库

| 方案 | 说明 | 推荐度 |
|------|------|--------|
| PostgreSQL 16 | 成熟、事务强、JSON 支持 | ⭐⭐⭐⭐⭐ |
| MySQL 8 | 兼容性广 | ⭐⭐⭐⭐ |
| SQLite | 嵌入，测试/轻量场景 | ⭐⭐⭐ |

**决策：PostgreSQL（Docker 运行，内存占用约 200 MB，可接受）。**

---

### 2.6 缓存

| 方案 | 说明 | 推荐度 |
|------|------|--------|
| Redis 7 | 必选，消息队列 + 缓存 + 会话 | ⭐⭐⭐⭐⭐ |
| go-cache（内存） | 仅测试/单机 | ⭐⭐⭐ |

**决策：Redis 7（Docker 运行）。**

---

### 2.7 Web 框架

| 方案 | 说明 | 推荐度 |
|------|------|--------|
| Gin | 性能最强，中间件生态丰富 | ⭐⭐⭐⭐⭐ |
| Echo | 类似 Gin，文档更简洁 | ⭐⭐⭐⭐ |
| Fiber | 性能最高，但稳定性略差 | ⭐⭐⭐ |

**决策：Gin。**

---

### 2.8 服务通信（微服务）

| 方案 | 内存占用 | 推荐度 | 说明 |
|------|---------|--------|------|
| gRPC + Protobuf | ~50 MB | ⭐⭐⭐⭐ | 性能高，跨语言，本环境可跑 |
| HTTP/JSON | 0 | ⭐⭐⭐⭐⭐ | 简单场景首选，开发快 |
| 共享内存（单进程） | 0 | ⭐⭐⭐⭐⭐ | Phase 1-2 最佳选择 |

**决策：单进程多模块（Phase 1-2）→ HTTP/gRPC（Phase 3+）。**

## 3. 最终推荐技术栈

### 阶段一（Week 1-2）：基础骨架

| 组件 | 选型 | 内存占用 |
|------|------|---------|
| 语言/框架 | Go 1.22 + Gin | ~50 MB |
| 数据库 | PostgreSQL 16 (Docker) | ~200 MB |
| 缓存 | Redis 7 (Docker) | ~80 MB |
| ORM | GORM | - |
| 配置 | Viper | - |
| 认证 | JWT + bcrypt | - |
| 容器 | Docker + Docker Compose | - |

**总计：~330 MB**，富余。

---

### 阶段二（Week 3-4）：IM 核心

| 组件 | 选型 | 内存占用 |
|------|------|---------|
| WebSocket | gorilla/websocket | - |
| 长连接管理 | 自研 Hub（内存） | ~100 MB |
| 消息推送 | WebSocket Hub + Redis Streams 抽象 | ~50 MB |
| 会话存储 | PostgreSQL + Redis | - |

**总计：~500 MB**，无压力。

---

### 阶段三（Week 5-8）：生产级扩展

| 组件 | 生产级 | 轻量化替代（本环境） | 内存节省 |
|------|--------|---------------------|---------|
| 消息队列 | Kafka | **Redis Streams** | ~1.5 GB |
| 全文搜索 | Elasticsearch | **Bleve**（嵌入） | ~1.5 GB |
| 文件存储 | MinIO/OSS | **MinIO + 本地 FS 双模式** | 可按环境取舍 |
| 服务通信 | gRPC | **HTTP + 单进程模块** | ~50 MB |

**调整后总计：~1 GB**，比生产级方案节省约 4 GB。

---

### 阶段四（Week 9-12）：高级特性

| 组件 | 选型 | 说明 |
|------|------|------|
| 音视频 | WebRTC（coturn TURN） | 可选，需 STUN/TURN 服务器 |
| 文档协作 | Yjs/CRDT 优先，OT 可作为学习方案 | 与前端生态更贴近 |
| 开放平台 | 机器人 + Webhook | 纯 HTTP，易实现 |
| 监控 | Prometheus（端点） | 仅暴露 metrics 端点，本地分析 |
| CI | GitHub Actions | 当前只做构建、测试、lint；部署流水线后续接入 |

## 5. 组件替换路径（开发 → 生产）

| 组件 | 开发阶段 | 生产阶段 | 替换成本 |
|------|---------|---------|---------|
| 消息队列 | Redis Streams | Kafka | 换实现类，不改调用方 |
| 全文搜索 | Bleve | Elasticsearch | 换实现类，不改调用方 |
| 文件存储 | MinIO / 本地 FS | MinIO / OSS | 已通过 FileStore 接口隔离 |
| 服务通信 | 单进程模块 | gRPC 微服务 | 重构通信层，业务逻辑不变 |

**关键：所有存储层接口抽象，业务代码不感知底层实现。**

## 6. Docker Compose 开发环境

```yaml
version: '3.9'

services:
  postgres:
    image: postgres:16-alpine
    environment:
      POSTGRES_DB: workpal
      POSTGRES_USER: workpal
      POSTGRES_PASSWORD: workpal123
    volumes:
      - postgres_data:/var/lib/postgresql/data
    ports:
      - "5432:5432"
    memory: 512M

  redis:
    image: redis:7-alpine
    ports:
      - "6379:6379"
    memory: 256M

  minio:
    image: minio/minio:latest
    environment:
      MINIO_ROOT_USER: workpal
      MINIO_ROOT_PASSWORD: workpal123456
    command: server /data --console-address ":9001"
    ports:
      - "9000:9000"
      - "9001:9001"
    volumes:
      - minio_data:/data

volumes:
  postgres_data:
  minio_data:
```

**启动命令：**

```bash
docker compose -f docker/docker-compose.yaml up -d

# 验证
docker compose -f docker/docker-compose.yaml ps
```

## 7. Makefile 常用命令

```makefile
.PHONY: deps run test build docker-up docker-down lint clean

deps:
	go mod download
	go mod tidy

run:
	go run ./cmd/server

test:
	go test -v -race -cover ./...

lint:
	golangci-lint run ./...

docker-up:
	docker compose -f ../docker/docker-compose.yaml up -d

docker-down:
	docker compose -f ../docker/docker-compose.yaml down

build:
	CGO_ENABLED=0 go build -o server ./cmd/server
```
