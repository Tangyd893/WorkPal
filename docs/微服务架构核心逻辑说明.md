## 二、核心设计决策

### 决策 1：为什么 Gateway 必须独立部署而不仅仅是 Nginx 反向代理

**不是 Nginx 能做的事情就一定要用 Nginx 做。**

| 能力 | Nginx | 自建 Gateway |
|------|-------|-------------|
| 路由转发 | ✓ | ✓ |
| 服务发现（从 Redis 读取实例） | 需 Lua 脚本 | 原生实现 |
| 按服务维度的熔断器状态机 | 需第三方模块 | 原生实现 |
| 按请求幂等性的差异化重试 | 难实现 | 原生实现 |
| 服务目录 API（`/gateway/services`） | 无 | 暴露学习面 |
| 与微服务运行时统一语言 | 运维层 | 开发层 |

**结论：** Nginx 适合做边缘代理，但当你的 Gateway 需要理解"下游服务有哪些""熔断器状态是什么""哪些请求可以重试"时，自建 Gateway 让你把这些逻辑写在同一个 Go 进程里，而不是散落在 Nginx 配置和 Lua 脚本中。

---

### 决策 2：为什么 Redis 同时承担四种职责

这是面试中很容易被问到的点——"你们是不是把 Redis 当万能工具了？"

**当前 Redis 承担的角色：**

| 角色 | 实现方式 | 为什么可以这样做 |
|------|---------|----------------|
| 服务注册发现 | Hash + TTL 心跳 | 实例名 → 地址的简单映射，不需要强一致性 |
| IM 跨实例扇出 | Pub/Sub | 实时消息投递，允许偶尔丢失（WebSocket 本身不保证送达） |
| 异步索引链路 | Redis Streams + Consumer Group | 需要 at-least-once 投递和消费者组 |
| 缓存（可选） | `pkg/cache` 抽象 | 标准缓存用途 |

**核心原则：** 不同场景用了不同的 Redis 数据结构，不是把 Redis 当万能工具，而是让每种数据结构解决它最擅长的问题：

- **Hash** — 注册发现的 KV 查找
- **Pub/Sub** — 实时事件广播（无持久化需求）
- **Streams + Consumer Group** — 需要持久化和消费者进度的消息队列
- **String/Set** — 标准缓存

**面试延伸：** "如果 Redis 挂了怎么办？"

- 注册发现：Gateway 回退到静态配置，服务仍然可用
- IM 扇出：本机 Hub 仍然工作，只是跨实例消息无法投递（单机模式降级）
- Streams：outbox 中的事件不会被删除，worker 会在 Redis 恢复后重试投递

### 决策 3：为什么 IM 消息要做 Outbox 模式而不是直接发事件

**问题：** 在 HTTP 请求的处理线程里直接发 Redis Streams 事件，有什么问题？

```
// 错误做法（伪代码）
func SendMessage(msg) {
    db.Insert(msg)           // 第1步：写数据库
    redis.XADD(msg)          // 第2步：发事件 ← 如果这里失败？
    return success
}
```

如果第 2 步失败：
- 消息已经写入数据库（用户看到了）
- 但 Search Service 永远不会索引这条消息
- 用户搜索不到刚发的消息 → 数据不一致

**Outbox 模式的做法：**

```
// 正确做法（伪代码）
func SendMessage(msg) {
    tx = db.Begin()
    tx.Insert(msg)              // 第1步：写消息
    tx.Insert(outbox_event)     // 第2步：写待投递事件 ← 同一事务！
    tx.Commit()                 // 两者同时成功或同时失败
    return success
}

// 后台 worker（独立 goroutine）
func OutboxWorker() {
    for {
        events = db.FindPendingOutboxEvents()
        for event in events {
            redis.XADD(event)
            db.MarkDelivered(event)
        }
    }
}
```

**关键收益：**

1. **原子性：** 消息和待投递事件在同一事务中落库
2. **补偿投递：** worker 失败不影响请求线程，事件不会丢失
3. **最终一致性：** 搜索索引可能有短暂延迟，但最终一定一致

### 决策 4：为什么每个服务有独立数据库

**不是"数据库拆分"这么简单，而是"数据边界"的声明。**

| 服务 | 数据库 | 包含的表 |
|------|--------|---------|
| User Service | `workpal_user` | users, departments, employees |
| IM Service | `workpal_im` | conversations, conversation_members, messages, message_outbox, message_reads |
| File Service | `workpal_file` | files |
| Workspace Service | `workpal_workspace` | tasks, schedule_events |

**为什么不能共享一个数据库？**

1. **Schema 耦合：** 如果所有表在一个库里，IM 服务改了 `messages` 表结构，会影响 User Service 的部署节奏
2. **资源隔离：** 一个服务的慢查询不会拖垮其他服务
3. **部署独立性：** 可以单独对 IM 服务做数据库扩容
4. **团队边界：** 每个服务团队只需理解自己领域的表结构

**当前项目的取舍：** 虽然部署在同一台 PostgreSQL 实例上（本地开发方便），但已经做到了 Schema 级别的隔离。如果未来需要物理隔离，只需要改连接字符串。

### 决策 5：为什么用 Repository 接口模式

**这是 P0/P1 优化中引入的重要工程实践。**

```go
// service.go
type Repository interface {
    ListTasks(ctx context.Context, userID int64) ([]*model.Task, error)
    CreateTask(ctx context.Context, task *model.Task) error
    GetTask(ctx context.Context, userID, taskID int64) (*model.Task, error)
    // ...
}

type Service struct {
    repo Repository  // 依赖接口，不依赖具体实现
}
```

**为什么不用具体的 `*repo.Repo`？**

1. **可测试性：** 测试时注入内存 mock，无需真实数据库
2. **可替换性：** 未来换 ORM 或存储方案，只需实现新 Repository
3. **依赖方向：** Service 依赖接口（自己的包），Repo 实现接口（自己的包 → Service 的接口），符合依赖反转

**面试要点：** 这不是 Go 特有问题。Java/Spring 中用 `@Repository` + `@Autowired`，Go 中用显式接口。本质都是"高层模块不依赖低层模块，两者都依赖抽象"。

## 三、面试预演：技术提问与标准回答

> 以下问题是假设你以 WorkPal 项目为例进行技术面试时，面试官可能提出的问题。每个回答都以"我能说清楚做了什么 + 为什么这样做 + 还有什么不足"为结构。

---

### 问题 1：你这个 Gateway 是怎么设计的？和 Nginx 有什么区别？

**回答结构：**

> "我们的 Gateway 是用 Go 自建的，不是简单的反向代理。它有四个层面的职责：
>
> **第一层是路由目录：** 我们显式声明了 11 条路由，每条路由绑定到一个下游服务、指定路径匹配模式、超时时间、是否支持 WebSocket。这不是 Nginx 的 `location` 块——我们通过 `/gateway/routes` 端点暴露了完整的路由元数据，包括超时和 WebSocket 标识。
>
> **第二层是服务发现：** 下游服务启动后把自己的地址注册到 Redis Hash，每 10 秒刷新 TTL。Gateway 从 Redis 读取实例。如果 Redis 不可用，自动回退到静态配置。`/gateway/services` 端点会告诉你每个服务当前是 `redis` 模式还是 `static` 模式。
>
> **第三层是入口治理：** 实现了按客户端 IP 的令牌桶限流、按服务的超时控制、仅对 GET/HEAD/OPTIONS 做重试（因为它们是幂等的）、以及按服务维度维护的熔断器状态机（closed → open → half-open）。
>
> **第四层是可观测面：** `/health/live` 返回网关自身存活，`/health/ready` 去探测所有下游服务是否可达。还有一个 `/metrics` 端点输出 Prometheus 格式的指标。
>
> 和 Nginx 的关键区别是：Nginx 做反向代理很好，但当你需要理解"下游服务有几个实例""熔断器当前是什么状态""哪些请求可以安全重试"这些微服务治理逻辑时，自建 Gateway 让你把这些逻辑写在代码里而非配置文件和 Lua 脚本中。"

---

### 问题 2：你们的服务注册发现怎么做的？Redis 挂了怎么办？

**回答结构：**

> "我们用 Redis 做服务注册发现。每个服务启动后执行三步：注册自己的地址到 Redis Hash、启动一个 ticker 每 10 秒刷新 TTL、在进程退出时尝试注销。
>
> Redis 挂了的影响是分级的：
>
> - **Gateway 不受影响：** 我们在启动时缓存了静态配置作为回退。如果 Redis 不可用，Gateway 自动切到 `discovery_mode: static`，用配置文件中预定义的地址转发请求。用户无感知。
> - **新实例不可被发现：** 如果 Redis 挂的时候启动一个新实例，它无法注册，但这个场景在实际中很少见（Redis 和实例通常不同时挂）。
> - **IM 跨实例扇出失效：** Redis Pub/Sub 不可用时，消息只能投递给本机 WebSocket 客户端。这会导致多实例部署时的消息漏投，但单机部署不受影响。
>
> 这个设计的取舍是：优先保证可用性（静态回退），而不是强一致性（完整注册中心）。对于学习型项目来说，这个权衡是合理的。如果到了生产环境，我会引入 Consul 或 Nacos 来做真正的健康检查和实例管理。"

---

### 问题 3：你们 IM 服务的 WebSocket 是怎么处理多实例部署的？

**回答结构：**

> "早期的 WebSocket Hub 只能在单实例内工作：本机的连接只认识本机的客户端。我们引入了 `ClusterBroadcaster` 来解决多实例问题。
>
> 核心思路是两步：
>
> **第一步，本机投递：** 消息到达后，先通过 Hub 把消息推给本机上属于该会话的在线客户端。这是最快路径。
>
> **第二步，跨实例扇出：** 同时把消息事件发布到 Redis Pub/Sub 的一个 channel。其他 IM 服务实例订阅了这个 channel，收到事件后在本机做同样的投递。
>
> 这里有一个关键设计：我们投递的是'用户级事件'而不是'房间级事件'。也就是说，消息体里包含目标用户 ID 列表，每个实例拿到后检查自己的在线用户是否在列表中。这避免了'所有实例都给所有房间广播'的浪费。
>
> 还有一个重要的降级策略：如果 Redis 不可用，跨实例扇出失效，但本机投递仍然工作。这意味着在单实例部署或所有用户连到同一实例的情况下，消息投递完全不受影响。"

---

### 问题 4：说一下 Outbox 模式——为什么不能直接在请求线程里发事件？

**回答结构：**

> "Outbox 模式解决的是'双写一致性问题'——你既要写数据库（消息持久化），又要发事件（通知搜索服务更新索引）。如果这两个操作不在同一事务中，就会出现一种情况：消息写入了数据库但事件没发出去，用户能在聊天记录里看到消息但搜索不到。
>
> 我们的做法是：
>
> 1. 在处理消息写入的同一数据库事务里，额外写一条 `message_outbox` 记录，状态为 `pending`
> 2. 事务提交后，请求线程就返回了——它不负责发事件
> 3. 后台有一个独立的 outbox worker，轮询状态为 `pending` 的记录
> 4. Worker 把事件发布到 Redis Streams，然后更新 outbox 记录为 `delivered`
>
> 这个设计的好处是：
> - **原子性：** 事件和业务数据一定同时落库或同时回滚
> - **解耦：** 发送事件的耗时（Redis 网络 IO）不影响 HTTP 请求的响应时间
> - **补偿：** 如果 Redis 当时不可用，worker 下次轮询时会重试（有 `retry_count` 和 `next_attempt_at` 字段做退避）
>
> 和直接发事件的本质区别：直接发事件是'尽力而为'，outbox 是'至少一次'。在我们的场景（消息→搜索索引）中，'至少一次'是正确的一致性级别。"

---

### 问题 5：为什么每个微服务有独立的数据库？你不觉得过度设计吗？

**回答结构：**

> "对于 5 个服务的项目，放在同一个 PostgreSQL 实例里完全合理。我们做的是 Schema 级别的隔离——每个服务有自己的数据库，表不交叉。这不是'过度设计'，而是'方向正确、实现务实'。
>
> Schema 隔离带来了几个实际好处：
>
> 1. **清晰的边界：** 新加入的开发者看 `workpal_im` 就知道哪些表属于 IM 服务
> 2. **独立的迁移：** 每个服务有自己的 `migrations/` 目录，互不干扰
> 3. **未来扩展路径：** 如果某个服务数据量暴涨（比如消息表很大），拆到独立 PostgreSQL 实例只需改连接字符串
>
> 如果面试官追问'为什么不一个库一把梭'：共享数据库最大的问题不是性能，而是**耦合**。当所有服务共享一个数据库，Schema 变更互相影响、慢查询互相拖累、团队之间需要协调部署节奏。Schema 隔离在物理上仍然共享实例，但在逻辑上已经划清了边界。"

### 问题 6：你们引入版本化迁移（golang-migrate）替代 AutoMigrate，为什么？

**回答结构：**

> "GORM 的 AutoMigrate 在开发阶段很方便——启动时自动建表、自动加列。但在真实项目中它有四个致命问题：
>
> 1. **不可回溯：** 你不知道数据库经过了哪些变更，没有版本历史
> 2. **不可回滚：** 如果加了列后发现有问题，AutoMigrate 不会帮你 DROP
> 3. **不可审计：** 团队协作时无法对 Schema 变更做 Code Review
> 4. **能力有限：** AutoMigrate 只能加列，不能做复杂的 ALTER、数据迁移、索引重建
>
> 我们引入 `golang-migrate`，每个迁移包含两个 SQL 文件：`up.sql`（前滚）和 `down.sql`（回滚）。迁移通过 Makefile 管理：
>
> ```bash
> make migrate-up SERVICE=im-service     # 执行未应用的迁移
> make migrate-down SERVICE=im-service   # 回滚最近一次
> make migrate-create SERVICE=im-service NAME=add_read_receipt  # 创建新迁移
> ```
>
> 关键是：我们不是'废除 AutoMigrate'，而是'保留 AutoMigrate 给开发阶段，正式环境用迁移'。这是一种务实的双轨策略。"

### 问题 7：你们前端怎么处理和后端 API 的契约？

**回答结构：**

> "前端和后端的 API 契约体现在三个层面：
>
> **第一层，API 客户端层（`frontend/src/api/workpal.ts`）：** 这是一个显式的函数目录，每个后端接口对应一个函数。函数定义了请求参数类型和返回值类型。这不是自动生成的，但有明确的一一对应关系。
>
> **第二层，TypeScript 类型系统（`frontend/src/types/`）：** `chat.ts` 和 `workspace.ts` 定义了前端使用的数据结构。这些类型和后端的 JSON 结构保持一致。当后端改了字段名，TypeScript 编译器会在前端构建时抛出类型错误。
>
> **第三层，翻译文本（`frontend/src/i18n.ts`）：** 175+ 个 key 覆盖了所有 UI 文本，支持中英双语。接口通过 `AppTranslations` 接口约束每个模块的翻译结构。
>
> 当前不足：我们还没有用 OpenAPI/Swagger 做自动化的契约生成和验证。如果要补上，会用 `swaggo/swag` 基于 Gin 注释生成 OpenAPI 文档，然后在前端 CI 中用 `openapi-typescript` 自动生成类型，实现真正的'契约驱动'。"

### 问题 8：你的熔断器具体怎么实现的？什么条件会触发熔断？

**回答结构：**

> "我们的熔断器是一个按服务维度维护的状态机，有三种状态：
>
> - **Closed（正常）：** 请求正常通过。在滑动窗口内统计失败次数。
> - **Open（熔断）：** 当窗口内失败次数达到阈值（默认 5 次），状态切换为 Open。新请求直接返回 503，不转发到下游。
> - **Half-Open（探测）：** 进入 Open 状态后等待一段时间（默认 30 秒），然后切换到 Half-Open，允许少量请求通过探路。如果这些请求成功，切回 Closed；如果失败，回到 Open 继续等待。
>
> 关键设计细节：
>
> 1. **按服务维度：** 不是全站共用一个断路器，而是每个下游服务独立维护。User Service 挂了不会触发对 IM Service 的熔断。
> 2. **只计 5xx 错误：** 4xx（客户端错误）不算失败，因为那是请求本身的问题，不是服务的问题。
> 3. **线程安全：** 使用 `sync.RWMutex` 保护状态读写。
> 4. **可观测：** 通过 `/gateway/services` 暴露每个服务的熔断器状态。
>
> 代码位置：`backend/cmd/gateway/breaker.go`，测试覆盖了状态转换的完整路径。"

### 问题 9：你的 Repository 接口模式有什么好处？为什么不用 GORM 直接操作？

**回答结构：**

> "Repository 接口模式的核心价值是**依赖反转**。Service 层依赖接口（定义在自己包里），Repo 层实现接口。
>
> 在我们这个项目中，这带来了两个直接好处：
>
> **1. 可测试性：** 测试 Service 时不依赖真实数据库。我们用内存 Mock 实现了所有 Repository 接口的方法——用 map 模拟表、用条件过滤模拟 SQL。结果是 Workspace Service 的 26 个单元测试全部在毫秒级别跑完，不需要启动 PostgreSQL。
>
> **2. 可替换性：** 未来如果要把 GORM 换成 sqlx、或者换一种存储方案（如 MongoDB），只需要写一个新的 Repository 实现，Service 层代码不变。
>
> 代码示例：
>
> ```go
> // service.go: 定义接口
> type Repository interface {
>     ListTasks(ctx, userID) ([]*model.Task, error)
>     CreateTask(ctx, task) error
> }
>
> type Service struct { repo Repository }
> func NewService(repo Repository) *Service { ... }
>
> // service_test.go: Mock 实现
> type mockRepo struct { tasks []*model.Task }
> func (m *mockRepo) ListTasks(ctx, userID) ([]*model.Task, error) { ... }
> func TestListTasks(t) { svc := NewService(&mockRepo{...}) }
>
> // main.go: 真实实现
> svc := service.NewService(repo.NewRepo(db))
> ```
>
> 这和 Spring 的 `@Autowired` + `@Repository` 本质一样——都是通过接口解耦。"

### 问题 10：如果让你重新设计这个项目，你会改什么？

**回答结构：**

> "我会改三件事：
>
> **第一，尽早引入接口：** Workspace Service 最初直接用 `*repo.Repo` 而不是 `Repository` 接口，导致它的测试覆盖为零——直到我们重构引入接口后才写得出来。这个应该在服务创建的第一天就定义好接口。
>
> **第二，API 文档先行：** 目前前后端接口没有自动化的契约文档。如果重来，我会在定义好 Handler 之后马上用 swaggo 生成 OpenAPI，然后基于它生成前端 TypeScript 类型。这样前端开发者不需要翻 Go 代码。
>
> **第三，迁移从第一天开始：** AutoMigrate 在开发初期很方便，但它让我们推迟了版本化迁移的引入。如果从第一个表就开始用迁移，后面的工程化文档会更加完备。
>
> 这三个问题本质上都是同一个教训：**工程化基础设施（接口抽象、API 文档、数据库迁移）不应该等到项目'做完了'才补，而是应该在写下第一行业务代码之前就准备就绪。**"
