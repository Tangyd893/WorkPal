# WorkPal 学习手册

这份手册是给“想借这个项目系统学习”的人准备的。它不会把所有细节一股脑堆上来，而是告诉你现在最值得看什么、为什么看、看到哪里算真正看懂。

## 1. 先建立整体认识

先用一句话抓住项目本质：

> WorkPal 是一个前端工作台 + 后端微服务 + 搜索异步链路的办公协作平台示例项目。

它不是：

- 单纯聊天 Demo
- 只有静态页面的前端练手项目
- 只有微服务外壳却仍靠单体入口运行的仓库

它现在已经具备这些真实结构：

- 前端工作台
- API Gateway
- User / IM / File / Search / Workspace 五个领域服务
- 服务自有数据库
- Redis Streams 异步事件
- Bleve 搜索索引

## 2. 建议学习顺序

如果你是第一次读这个仓库，建议按下面顺序来：

1. 跑起来
2. 看前端页面结构
3. 看前端 API 调用去了哪里
4. 看 Gateway 如何分发
5. 看每个服务如何启动和接数据库
6. 看 IM -> Search 的异步链路
7. 再回头评价哪些设计做得好、哪些还可以继续优化

这个顺序的好处是：你先理解系统怎么活着，再理解代码为什么这样写。

## 3. 第一阶段：先把项目跑通

先读：

- `README.md`
- `README-cn.md`
- `docs/测试手册.md`

跑通之后，建议你至少亲手验证一次：

- `admin / admin123` 登录
- 总览跳转
- 通讯录搜索
- 创建私聊和群聊
- 任务创建
- 日程创建
- 文件上传

如果这一步没做，后面读代码容易飘在空中。

## 4. 第二阶段：理解前端工作台

### 4.1 从页面骨架开始

优先看：

- `frontend/src/App.tsx`
- `frontend/src/pages/WorkspacePage.tsx`

你要先看明白两件事：

1. 路由如何组织
2. 各板块如何被挂进工作台

当前工作台板块顺序是：

- 总览
- 沟通
- 任务
- 日程
- 文件
- 通讯录

### 4.2 再看状态管理

接着看：

- `frontend/src/hooks/useAuthStore.ts`
- `frontend/src/hooks/usePreferencesStore.ts`
- `frontend/src/hooks/useChatController.ts`

这里最值得学的是状态分层：

- 鉴权状态单独管理
- 偏好设置单独管理
- 聊天运行时状态由控制器组织

这比把所有东西都塞进页面组件里要成熟得多。

### 4.3 最后看 API 封装

看：

- `frontend/src/api/request.ts`
- `frontend/src/api/workpal.ts`

你需要弄清楚：

- 前端和后端的响应契约是什么
- 任务、日程、文件、沟通分别调用哪些接口
- 前端如何把页面动作映射成 API 操作

如果这一层看懂了，你会开始意识到：前端不是“展示层”，而是产品工作流的真正编排层。

## 5. 第三阶段：理解后端微服务是怎么落地的

### 5.1 从 Gateway 开始

先看：

- `backend/cmd/gateway/main.go`

重点关注：

- 路径路由规则
- 请求 ID
- 限流
- 聚合健康检查

你要问自己两个问题：

1. 为什么前端只访问 `8080`？
2. 为什么网关的 `/health` 要检查下游服务？

如果这两个问题答清楚了，你就理解了“统一入口”的真正意义。

### 5.2 再看公共平台层

看：

- `backend/configs/config.go`
- `backend/internal/platform/runtime.go`

这部分非常关键，因为它决定了每个服务“如何被启动出来”。

这里要重点理解：

- 配置读取
- 服务自有数据库名
- 数据库自动创建
- 通用健康检查注册

这是从“模块化项目”走向“微服务项目”的真正骨架。

### 5.3 然后分服务看

建议顺序如下。

#### User Service

先看：

- `backend/cmd/user-service/main.go`
- `backend/internal/platform/devseed.go`
- `backend/internal/user/*`

学习重点：

- 登录流程
- 用户与员工档案模型
- 开发种子数据如何补齐
- 通讯录搜索如何做模糊匹配和部门筛选

#### IM Service

看：

- `backend/cmd/im-service/main.go`
- `backend/internal/im/model`
- `backend/internal/im/repo`
- `backend/internal/im/service`
- `backend/internal/im/handler`
- `backend/internal/im/ws`

学习重点：

- 会话和消息的领域模型
- 私聊、群聊、群公告
- WebSocket 连接管理
- 为什么消息写入要落在 IM Service

#### File Service

看：

- `backend/cmd/file-service/main.go`
- `backend/internal/file/*`
- `backend/internal/clients/im.go`

学习重点：

- 文件元数据和文件内容分离
- MinIO 与本地存储抽象
- 文件服务为什么还要向 IM Service 校验群成员关系

#### Search Service

看：

- `backend/cmd/search-service/main.go`
- `backend/internal/search/*`
- `backend/internal/events/messages.go`
- `backend/pkg/msgqueue/redis_streams.go`

学习重点：

- 为什么搜索服务不直接写业务主库
- 为什么要从 Redis Streams 消费消息事件
- 为什么当前用 Bleve 而不是更重的搜索集群

#### Workspace Service

看：

- `backend/cmd/workspace-service/main.go`
- `backend/internal/workspace/*`

学习重点：

- 任务和日程为什么独立成服务
- CRUD 与“分享次数”这类轻协作行为如何建模

## 6. 第四阶段：把一条真实链路看穿

这里推荐你完整追一遍“发一条群消息”的路径。

顺序如下：

1. 前端聊天输入框提交
2. `workpalApi.sendMessage`
3. Gateway 路由到 IM Service
4. IM Service 写入 `workpal_im`
5. IM Service 发布 Redis Streams 事件
6. Search Service 消费事件并写 Bleve 索引
7. WebSocket 把新消息推给在线成员

如果你能把这条链路从头到尾说清楚，这个项目最难的一层你就已经过了。

## 7. 这个项目最值得学的地方

### 7.1 它已经不是“假微服务”

很多项目只是：

- 多几个目录
- 多几个 main
- 但仍共用一个单体入口或一套混乱的共享数据

这个项目现在已经做到：

- 服务独立启动
- 服务边界清晰
- 数据库按服务拆分
- 网关成为真实入口

这很值得学习。

### 7.2 它把“同步调用”和“异步事件”都做进去了

你可以同时学到：

- File / Search 服务如何通过 HTTP 和其他服务协作
- IM / Search 如何通过事件链路解耦

这比只会写 CRUD 项目更有系统设计价值。

### 7.3 它前后端是连着想的

前端并不是随便堆几个页面，而是明确围绕：

- 总览
- 沟通
- 任务
- 日程
- 文件
- 通讯录

这些办公协作场景组织起来的。

## 8. 当前还不够完美的地方

学习时也要会辨别“不足在哪里”，而不是把现状神化。

### 8.1 WebSocket 还不是分布式连接层

当前 Hub 仍在单进程内存里，这意味着：

- 单实例足够
- 多实例扩展仍需继续设计

### 8.2 Gateway 还是轻量实现

它已经有真实职责，但还没进入更完整的生产治理阶段，例如：

- 熔断
- 更细粒度超时策略
- 更强的观测性

### 8.3 Search 仍是轻量实现

Bleve 很适合本地和教学，但在更大规模生产场景下通常还会继续升级。

## 9. 你可以怎么继续练

如果你想把这个仓库当练习场，推荐按下面顺序加练习题：

1. 给 Gateway 增加更细粒度的服务级超时配置
2. 给 User Service 增加内部用户摘要接口，减少未来跨服务重复查询
3. 给 IM Service 的 WebSocket 做多实例会话同步设计
4. 给 Search Service 做全量重建索引命令
5. 给 Workspace Service 增加更完整的任务字段和筛选能力

## 10. 一条真正有用的阅读建议

不要一开始就试图“把所有代码看完”。

更有效的方法是：

1. 从一个已知页面出发
2. 顺着一次真实操作往后追
3. 遇到服务边界时停下来总结
4. 再切换到下一条链路

例如你可以分四天练：

- 第一天：登录 + 通讯录
- 第二天：私聊/群聊 + WebSocket
- 第三天：文件 + 权限校验
- 第四天：任务/日程 + 数据归属

这样吸收速度会快很多。

## 11. 结语

这份项目最适合作为“从业务代码走向架构理解”的桥梁。

如果你认真把这四层吃透：

- 前端工作台
- Gateway
- 领域服务
- 事件链路

你学到的就不只是某几个接口怎么写，而是一个协作平台为什么会长成现在这样。
