# WorkPal 学习手册

这份手册面向"想通过一个完整项目学习微服务、网关治理、实时协作和前后端边界"的读者。它不追求一次讲完全部代码，而是帮你建立稳定的阅读顺序和判断框架。

## 1. 先用一句话看懂项目

> WorkPal 是一个由"前端工作台 + API Gateway + 多个领域微服务 + Redis 驱动的基础设施能力"组成的办公协作平台。

当前最值得学习的，不是单个 CRUD，而是这些能力如何拼在一起：

- Gateway 作为统一入口，承担路由、限流、重试、熔断、服务发现、搜索降级
- OpenTelemetry 全链路追踪 + 结构化 JSON 日志 + Prometheus/Grafana 监控看板
- Redis 同时承担服务注册发现、IM 扇出、Redis Streams 异步链路
- 各服务各管各的数据边界，各管各的数据库
- 消息冷热分离存储（近期写扩散 / 历史读扩散）
- 消息发送幂等保护（幂等令牌去重）+ 搜索服务熔断降级
- 敏感操作审计日志记录
- 任务创建 Saga 编排（带补偿回滚的事务状态机）
- Kubernetes 部署清单（Deployment + Service + 探针 + 滚动更新）
- 前端工作台通过网关整合多个领域能力
- 消息支持编辑、撤回、行内编辑
- 版本化数据库迁移体系覆盖四个领域服务的多轮迁移

## 3. 用 Spring Cloud Alibaba 视角来映射

这不是把 Java 组件名字硬搬到 Go 里，而是借助成熟微服务视角来读当前实现。

| 学习视角 | 当前 WorkPal 实现 | 你应该重点看什么 |
| --- | --- | --- |
| Spring Cloud Gateway | `backend/cmd/gateway` | 路由、限流、重试、熔断、搜索降级、健康检查 |
| Nacos | `backend/internal/platform/registry.go` + `/gateway/services` | 服务注册、服务发现、静态回退 |
| Sentinel | Gateway 的限流、超时、重试、熔断、就绪检查 | 入口层治理策略 |
| Feign | `backend/internal/clients/*` | 服务间 HTTP 调用封装 |
| RocketMQ | `message_outbox` + `backend/pkg/msgqueue/redis_streams.go` | IM 到 Search 的可靠异步索引链路 |
| Sleuth / Micrometer Tracing | `backend/internal/platform/observability.go` | Gateway trace 注入、OpenTelemetry SDK、Jaeger 链路 |

如果你学的是"Spring Cloud 五大组件"那条路线，也可以这样理解：

- Gateway -> WorkPal Gateway
- Nacos -> Redis 服务注册发现
- Ribbon -> Gateway 基于发现结果做实例选择
- Feign -> `internal/clients`
- Sentinel / Hystrix -> Gateway breaker / retry / rate limit
- RocketMQ -> Outbox + Redis Streams

你应该看懂两条链路。

### 8.1 用户定向消息链路

1. IM Service 处理消息写入
2. Handler 拿到会话成员列表
3. `ClusterBroadcaster.BroadcastUsers` 先投递本机在线用户
4. 再把事件发到 Redis Pub/Sub
5. 其他实例订阅后把消息推给各自本机在线用户

### 8.2 房间广播链路

1. WebSocket 客户端产生已读或已读全部等房间事件
2. Hub 先做本机房间广播
3. 再通过 cluster broadcaster 把房间事件扇出到其他实例
4. 其他实例对本机已加入该房间的客户端继续广播

### 8.3 消息编辑与撤回

前端新支持的消息操作：

1. 点击自己消息旁的"编辑"按钮进入行内编辑模式
2. 按 Enter 提交编辑，按 Escape 取消
3. 后端 `PUT /api/v1/messages/:id` 更新消息内容
4. 点击"撤回"按钮调用 `DELETE /api/v1/messages/:id`
5. 前端收到编辑/撤回的 API 响应后实时更新界面
6. IM 服务在消息变更时同步更新 `message_outbox`，触发搜索索引刷新

## 10. 第七阶段：看数据库迁移与工程化

这是 P1 优化中新增的工程化基础设施。

重点看：

- [migrations/](/D:/Code/codex/WorkPal/backend/migrations/) 目录下的四个服务迁移文件
- [Makefile](/D:/Code/codex/WorkPal/backend/Makefile) 中的 `migrate-*` 目标

你要看懂的关键点：

### 10.1 为什么需要版本化迁移

之前项目依赖 GORM 的 `AutoMigrate` 在启动时自动建表。这在开发阶段很方便，但在真实项目中存在几个问题：

1. **不可回溯：** 你无法知道数据库经过了哪些变更
2. **不可回滚：** 出问题时无法快速回到上一版本
3. **不可审计：** 团队协作时无法对 schema 变更做 code review
4. **不精确：** AutoMigrate 只能添加列，不能做复杂的迁移操作

### 10.2 golang-migrate 的使用方式

```powershell
# 安装 CLI 工具
make migrate-install

# 运行所有未执行的迁移
make migrate-up SERVICE=user-service

# 回滚最近一次迁移
make migrate-down SERVICE=im-service

# 创建新的迁移文件
make migrate-create SERVICE=workspace-service NAME=add_task_labels
```

每个迁移包含两个 SQL 文件：

- `NNNN_name.up.sql` — 前滚（创建表、添加列等）
- `NNNN_name.down.sql` — 回滚（对应的 DROP 操作）

### 10.3 迁移与 AutoMigrate 的关系

当前项目两种方式都可以使用：

- **快速开发：** 直接启动服务，AutoMigrate 自动建表
- **正式部署：** 先运行迁移，再启动服务

建议在 CI 和正式环境中优先使用迁移，确保 schema 变更有迹可循。

## 10. 第十阶段：看消息推送性能优化

这是从"基本可用"到"高吞吐低成本"的进阶。消息冷热分离 + Redis Streams 消费者组是写扩散性能问题的经典解法。

重点看：

- [message_svc.go](/D:/Code/codex/WorkPal/backend/internal/im/service/message_svc.go)
- [message_repo.go](/D:/Code/codex/WorkPal/backend/internal/im/repo/message_repo.go)
- [model.go](/D:/Code/codex/WorkPal/backend/internal/im/model/model.go)
- [backend/migrations/im-service/002_message_idempotency_and_tier.up.sql](/D:/Code/codex/WorkPal/backend/migrations/im-service/002_message_idempotency_and_tier.up.sql)
- [testing/performance/群发消息压测.js](/D:/Code/codex/WorkPal/testing/performance/群发消息压测.js)

你应该看懂这几件事：

### 10.1 写扩散的性能瓶颈

1. 传统模式：用户在大群发消息 → 查询群成员列表 → 向每个人写入一条收件箱记录
2. 2000 人大群：发一条消息 = 2000 次 INSERT
3. 瓶颈：DB 写入量线性增长，事务时间随群人数线性增长

### 10.2 冷热分离方案

1. **热区（近期消息）**：最近 7 天消息走写扩散，写入成员收件箱表
2. **冷区（历史消息）**：超过 7 天的消息走读扩散，直接按 `conversation_id + time_range` 查询
3. 消息表增加冷热分区索引，冷区数据可迁移至低成本存储

### 10.3 Redis Streams 消费者组

1. 替换原 Pub/Sub 模式（消息不持久，消费者离线会丢消息）
2. 消费者组支持 ACK 确认、消息重试、多消费者负载均衡
3. Search Service 消费索引事件后 ACK，未 ACK 的事件可被其他消费者重新消费

### 10.4 压测验证

1. 压测脚本模拟 2000 人大群发消息场景
2. 输出 P50/P99 延迟、吞吐量、DB 写入量对比
3. 目标：群发延迟降低 80%+，DB 写入量减少 70%+

参考文档：[消息推送性能优化报告](/D:/Code/codex/WorkPal/docs/消息推送性能优化报告.md)

## 11. 第十一阶段：看幂等与韧性

生产环境的核心保障：客户端重试不能导致重复操作，下游故障不能让上游崩掉。

重点看：

- [message_handler.go](/D:/Code/codex/WorkPal/backend/internal/im/handler/message_handler.go) — 幂等令牌接收
- [message_svc.go](/D:/Code/codex/WorkPal/backend/internal/im/service/message_svc.go) — 幂等去重逻辑
- [backend/migrations/im-service/002_message_idempotency_and_tier.up.sql](/D:/Code/codex/WorkPal/backend/migrations/im-service/002_message_idempotency_and_tier.up.sql) — 幂等令牌表
- [gateway/app.go](/D:/Code/codex/WorkPal/backend/cmd/gateway/app.go) — 搜索路由熔断配置

你应该看懂这几件事：

### 11.1 消息发送幂等

1. 前端生成 `idempotency_key`（UUID），随请求发送
2. 后端以 `(conversation_id, sender_id, idempotency_key)` 唯一约束去重
3. 令牌有效期 5 分钟，定时清理过期令牌
4. 客户端网络问题重发请求时，后端直接返回已存在的消息 ID

### 11.2 搜索服务降级

1. Gateway 对 `/api/v1/search` 路由启用熔断器
2. 连续失败 5 次 → 熔断打开（直接返回错误，不调用下游）
3. 30 秒后半开探测，成功则关闭熔断
4. 搜索不可用时前端展示提示，**消息收发完全不受影响**——这是降级的关键设计原则

参考文档：[韧性设计与降级策略](/D:/Code/codex/WorkPal/docs/韧性设计与降级策略.md)

## 13. 第十三阶段：看 Saga 分布式事务

微服务跨库操作的最终一致性方案。Saga 不是替代 ACID，而是"跨服务时你只能选最终一致"的工程解决方案。

重点看：

- [backend/migrations/workspace-service/003_task_sagas.up.sql](/D:/Code/codex/WorkPal/backend/migrations/workspace-service/003_task_sagas.up.sql)
- [backend/internal/workspace/service/service.go](/D:/Code/codex/WorkPal/backend/internal/workspace/service/service.go)
- [backend/internal/workspace/handler/handler.go](/D:/Code/codex/WorkPal/backend/internal/workspace/handler/handler.go)

你应该看懂这几件事：

### 13.1 场景："创建带提醒的任务"

1. Workspace Service 写任务 → 写日程提醒（两个服务，两个数据库）
2. 如果步骤 2 失败，步骤 1 需要回滚
3. 不能用数据库事务（跨库），只能用 Saga

### 13.2 编排式 Saga 实现

1. `task_sagas` 表记录事务状态：`pending → running → success / failed / compensating`
2. 正向步骤成功 → 标记 `success`
3. 任一正向步骤失败 → 后台补偿 worker 调用补偿接口回滚已执行的步骤
4. 补偿步骤必须幂等（可能被重复调用）

### 13.3 为什么选非核心链路

Saga 会给系统增加复杂度（状态表、补偿逻辑、幂等要求）。选"任务创建"而非"消息发送"作为试点，是因为：
- 任务创建频率低，事务冲突少
- 失败后果可控（任务没创建成功 vs. 消息丢了）
- 验证方案可行性后再推广到核心链路

参考文档：[分布式事务设计](/D:/Code/codex/WorkPal/docs/分布式事务设计.md)

## 14. 第十四阶段：看 K8s 部署

Docker Compose 适合本地开发，Kubernetes 才是生产部署的标准答案。

重点看：

- [docker/kubernetes/工作负载.yaml](/D:/Code/codex/WorkPal/docker/kubernetes/工作负载.yaml)
- [backend/Dockerfile](/D:/Code/codex/WorkPal/backend/Dockerfile)

你应该看懂这几件事：

### 14.1 K8s 清单结构

每个微服务都包含：
- **Deployment**：副本数、资源 requests/limits、滚动更新策略
- **Service**：ClusterIP 暴露，k8s DNS 做服务发现
- **ConfigMap**：非敏感的配置参数
- **readinessProbe / livenessProbe**：HTTP GET `/health` 和 `/health/live`
- **滚动更新**：`maxUnavailable: 1` + `maxSurge: 1`，确保更新期间服务不断流

### 14.2 K8s 下的服务发现变化

1. 在 k8s 中，Service 的 DNS 名（如 `user-service.workpal.svc.cluster.local`）天然替代了 Redis 注册发现
2. Gateway 的 `upstreamService` 可以直接指向 k8s Service DNS
3. 但离开 k8s 环境时（如多云/混合部署），Consul/Nacos 仍有价值

### 14.3 部署命令

```bash
kubectl apply -f docker/kubernetes/工作负载.yaml
kubectl -n workpal get pods
kubectl -n workpal rollout undo deployment/gateway  # 回滚
```

## 16. 当前项目的学习价值与边界

### 已经很值得学的部分

- 微服务入口层治理（路由、限流、熔断、重试、搜索降级）
- 轻量服务注册发现（Redis + 静态回退）
- Redis 同时承担多类基础设施职责
- 实时系统从单实例向多实例演进（Pub/Sub 扇出）
- Outbox + Redis Streams 消费者组的最终一致性异步链路
- Repository 接口模式用于单元测试 mock
- 版本化数据库迁移体系（含幂等表、审计表、Saga 表）
- OpenTelemetry 全链路追踪 + zerolog 结构化日志 + `/metrics` 端点
- 消息冷热分离存储 + 消息发送幂等保护
- 搜索服务熔断降级 + 审计日志记录
- 任务创建 Saga 编排（补偿式分布式事务）
- Kubernetes 部署清单（Deployment + Service + 探针 + 滚动更新）

### 还没有完全做满的部分 —— 为什么这些值得继续做

这些不是缺陷，而是当前项目故意保留的工程化扩展方向。每一个方向背后都对应一个在真实项目中绕不开的工程问题。

> 注意：原本列出的 6 个方向中，**分布式链路追踪（OpenTelemetry）**、**结构化日志与日志聚合**、**容器编排（Kubernetes）** 已在本轮迭代中完成，详见第九、十、十四阶段。

#### 1. 独立配置中心

**当前状态：** 每个服务通过 Viper 读取本地 `config.yaml` 文件。已支持 `config.yaml` + 环境变量覆盖的优先级链，但配置修改仍需重启服务。

**为什么需要升级：**

- **配置漂移：** 当你把 6 个服务部署到 3 个环境（dev/staging/prod），每个环境有 6 份配置文件，配置项可能不一致。改一个数据库连接串要改 6 个文件、3 套环境 = 18 次操作。
- **热更新：** 本地配置修改后必须重启服务才能生效。真实场景中，修改限流阈值、熔断参数不应该导致服务重启。
- **审计：** 谁在什么时候改了什么配置项？本地文件无法回答这个问题。

**推荐方向：**

- 轻量方案：引入 Consul KV Store，Go 用 `hashicorp/consul/api` 监听配置变更
- 工业级方案：Nacos（Go SDK `nacos-group/nacos-sdk-go`）
- 学习重点：配置变更如何通过 channel 通知服务内部组件（如 breaker 重新设置阈值）

**与 Spring Cloud 的对应：** Spring Cloud Config / Nacos Config

#### 2. 完整注册中心

**当前状态：** 服务启动后注册到 Redis Hash，每 10 秒刷新 TTL。Gateway 从 Redis 读取实例列表。

**为什么需要升级：**

- **健康检查不足：** Redis TTL 只能判断实例是否还在心跳。一个实例可能心跳正常但 HTTP 接口已挂（如数据库连接池耗尽）。
- **权重和元数据：** 无法为实例设置权重（如新版本先 10% 流量灰度），无法携带元数据（版本号、机房、标签）。
- **服务治理：** 无法做流量染色、蓝绿部署、金丝雀发布。

**推荐方向：**

- 轻量方案：Consul（内置健康检查、DNS 接口、KV 存储）
- 工业级方案：Nacos（服务发现 + 配置中心一体）
- 学习重点：从 Redis 方案到 Consul 方案的迁移过程中，Gateway 的 `upstreamService` 结构如何平滑切换

**与 Spring Cloud 的对应：** Spring Cloud Alibaba Nacos Discovery

#### 3. API 文档自动生成

**当前状态：** 没有自动生成的 API 文档。要知道有哪些接口、参数、返回值，只能看代码。

**为什么需要升级：**

- **前后端协作：** 前端开发者不应该翻 Go 代码来找接口定义。一份 OpenAPI 文档可以直接导入 Postman 或生成前端 TypeScript 类型。
- **接口契约：** 当后端修改接口时，API 文档生成可以在 CI 中做 breaking change 检测。
- **新人上手：** 新来的后端开发者不需要逐个读 handler 代码就能理解所有接口。

**推荐方向：**

- 用 `swaggo/swag` 在 Gin handler 上加注释 → 自动生成 OpenAPI 3.0 JSON
- Gateway 的 `/docs` 端点挂载 Swagger UI
- 在 CI 中校验 API 文档与代码是否同步
- 学习重点：Go struct tag 和 swaggo 注释如何映射到 OpenAPI schema

**与 Spring Cloud 的对应：** SpringDoc OpenAPI（原 SpringFox）

#### 为什么它们被故意留在"没做满"

这 3 个方向各有其学习价值，但它们本身也是独立的工程主题。当前项目选择：

- 先把微服务核心链路 + 可观测性 + 性能 + 韧性做扎实
- 再把这些治理层能力作为自然扩展的方向
- 让每个方向都可以独立学习，不必一次性堆满

如果你正在准备微服务面试，把已完成的 12 个方向的技术原理和业务价值讲清楚，再加上这 3 个方向的设计思路，比"全都做了但都做不深"更有说服力。
