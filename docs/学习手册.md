# WorkPal 学习手册

这份手册面向“想通过这个项目系统学习微服务和协作平台实现”的读者。它不追求把所有代码一次性讲完，而是帮你建立正确的阅读顺序、抓住最值得学的结构，并把当前 Go 实现和你熟悉的 Spring Cloud Alibaba 思路对应起来。

## 1. 先用一句话看懂项目

> WorkPal 是一个前端工作台 + API Gateway + 领域微服务 + 异步搜索链路组成的办公协作平台示例项目。

它现在已经不是：

- 单纯聊天 Demo
- “目录上看着像微服务，运行上还是单体”的项目
- 只有页面没有真实后端协作链路的展示仓库

它当前已经具备这些真实结构：

- 前端工作台
- API Gateway
- User / IM / File / Search / Workspace 五个领域服务
- 服务自有数据库
- Redis Streams 异步事件
- Bleve 搜索索引

## 2. 建议学习顺序

建议按下面的顺序走：

1. 先把项目跑通
2. 看前端工作台如何组织模块
3. 看前端 API 如何汇聚到网关
4. 把 Gateway 看透
5. 再分服务看领域边界
6. 最后追一条真实业务链路

这个顺序的好处是：你先理解系统怎么活着，再理解代码为什么这样写。

## 3. 如果你熟悉 Spring Cloud Alibaba，可以这样映射

这不是把 Java 组件名硬搬到 Go 里，而是借它的学习视角来读当前实现。

| Spring Cloud Alibaba 视角 | 当前 WorkPal 实现 | 你该看什么 |
| --- | --- | --- |
| Spring Cloud Gateway | `backend/cmd/gateway` | 路由转发、治理策略、管理面 |
| Nacos（服务目录 / 配置中心） | `backend/configs/*` + `/gateway/services` | 服务地址、超时、治理配置、下游服务目录 |
| Sentinel | Gateway 的限流、重试、熔断、就绪检查 | `rate_limit.go`、`transport.go`、`breaker.go` |
| OpenFeign / 服务间调用 | `backend/internal/clients/*` | File / Search 如何访问 IM Service |
| RocketMQ | Redis Streams | IM -> Search 的消息索引事件链路 |

这套映射很重要，因为它能帮你把“Go 项目里的微服务实现”放进一个更成熟的微服务认知框架里。

## 4. 第一阶段：先跑起来

先读：

- `README.md`
- `README-cn.md`
- `docs/测试手册.md`

至少亲手验证一次：

- `admin / admin123` 登录
- 总览跳转
- 通讯录搜索
- 创建私聊和群聊
- 任务创建
- 日程创建
- 文件上传

如果这一层没做，后面读代码会很容易停留在“知道文件在哪里”，但不知道系统为什么这么长。

## 5. 第二阶段：理解前端工作台

### 5.1 先看页面骨架

优先看：

- `frontend/src/App.tsx`
- `frontend/src/pages/WorkspacePage.tsx`

先回答两个问题：

1. 工作台有哪些板块？
2. 页面切换是怎样组织的？

当前工作台模块顺序是：

- 总览
- 沟通
- 任务
- 日程
- 文件
- 通讯录

### 5.2 再看状态管理

接着看：

- `frontend/src/hooks/useAuthStore.ts`
- `frontend/src/hooks/usePreferencesStore.ts`
- `frontend/src/hooks/useChatController.ts`

你会看到前端状态不是一锅粥，而是分成：

- 鉴权状态
- 偏好设置
- 聊天运行态

这正是协作产品前端应该有的分层。

### 5.3 最后看 API 封装

看：

- `frontend/src/api/request.ts`
- `frontend/src/api/workpal.ts`

要搞清楚：

- 前端统一响应契约是什么
- 页面动作如何映射成 API
- 哪些模块是真后端数据，哪些仍是展示型数据

## 6. 第三阶段：把 Gateway 看透

这是这次最值得学的一层，也是你之后读微服务项目时最需要具备的视角。

### 6.1 Gateway 现在不只是一个转发器

当前 Gateway 已经承担了这些职责：

- 前端统一入口
- 路由目录
- 下游服务目录
- 请求 ID 注入
- 限流
- 服务级超时
- 只对幂等读请求启用重试
- 熔断器
- 存活 / 就绪 / 聚合健康检查

也就是说，它已经不再是一个 `switch + ReverseProxy` 的演示壳。

### 6.2 阅读顺序

建议按这个顺序看 Gateway 代码：

1. `backend/cmd/gateway/main.go`
2. `backend/cmd/gateway/app.go`
3. `backend/cmd/gateway/rate_limit.go`
4. `backend/cmd/gateway/transport.go`
5. `backend/cmd/gateway/breaker.go`
6. `backend/cmd/gateway/gateway_test.go`

这个顺序对应的是：

- 入口
- 路由与服务目录
- 流量治理
- 上游重试
- 熔断策略
- 可验证的行为

### 6.3 你应该重点看什么

#### 路由目录

看 `app.go` 里的 `routeSpec` 和 `buildRouteSpecs`。

你要理解的是：

- 路由不再埋在 `switch` 里
- 每条路由带有服务归属、超时、用途说明
- `/gateway/routes` 把这些规则显式暴露出来，方便学习和排查

#### 服务目录

看 `upstreamService`、`buildUpstreamServices` 和 `/gateway/services`。

你要理解的是：

- 网关知道每个下游服务是谁
- 每个服务有自己的基础地址、健康地址、超时、重试次数、熔断状态
- 这相当于一个“轻量服务目录”，可以把它看作是静态版 Nacos 视角

#### 限流

看 `rate_limit.go`。

要点：

- 网关对入口流量做统一控制
- 健康检查和管理面接口不参与限流
- 这是最轻但很实用的入口流量治理

#### 重试

看 `transport.go`。

这里要注意它的取舍：

- 只对 `GET / HEAD / OPTIONS` 这类幂等读请求启用重试
- 写请求不重试
- 这是非常典型、也非常值得学习的微服务入口层保守策略

它的意义在于：

- 提高读请求面对瞬时上游波动时的成功率
- 避免对写请求造成重复提交风险

#### 熔断器

看 `breaker.go`。

你要看懂三个状态：

- `closed`
- `open`
- `half_open`

以及它们分别意味着什么：

- `closed`：正常放行
- `open`：短时间拒绝流量，保护下游
- `half_open`：试探性恢复

这正是你从 Sentinel / Resilience4j 等生态里应该学会的基础治理模型。

#### 健康检查

现在 Gateway 暴露三个层次：

- `/health/live`
- `/health/ready`
- `/health`

理解方式如下：

- `live`：进程还活着
- `ready`：网关以及下游服务当前可以接流量
- `health`：当前默认也输出聚合健康状态

这个区分在容器化和微服务环境里非常重要。

### 6.4 Gateway 这一层你真正该学到什么

不是“会写反向代理”，而是：

1. 入口层必须有路由目录
2. 入口层必须有治理策略
3. 入口层必须有可观测性和自描述能力
4. 入口层的重试和熔断必须保守而明确

如果你把这四点吃透了，读别的微服务项目时会有很强的判断力。

## 7. 第四阶段：分服务看领域边界

### 7.1 User Service

看：

- `backend/cmd/user-service/main.go`
- `backend/internal/platform/devseed.go`
- `backend/internal/user/*`

重点：

- 登录流程
- 目录搜索
- 部门与员工档案
- 开发种子数据补齐

### 7.2 IM Service

看：

- `backend/cmd/im-service/main.go`
- `backend/internal/im/*`

重点：

- 私聊、群聊、群公告
- 消息模型
- WebSocket 连接管理
- 为什么消息事实必须落在 IM Service

### 7.3 File Service

看：

- `backend/cmd/file-service/main.go`
- `backend/internal/file/*`
- `backend/internal/clients/im.go`

重点：

- 文件元数据与对象内容分离
- 通过 IM Service 进行群成员权限校验

### 7.4 Search Service

看：

- `backend/cmd/search-service/main.go`
- `backend/internal/search/*`
- `backend/pkg/msgqueue/redis_streams.go`

重点：

- 为什么它不直接写业务主库
- 为什么它靠消息事件更新索引
- 为什么这里用 Bleve 就够学

### 7.5 Workspace Service

看：

- `backend/cmd/workspace-service/main.go`
- `backend/internal/workspace/*`

重点：

- 为什么任务和日程值得独立成服务
- 如何给协作型业务建模轻量共享行为

## 8. 第五阶段：追一条真实链路

推荐你完整追一遍“发一条群消息”的路径：

1. 前端聊天输入框提交
2. `workpalApi.sendMessage`
3. 请求进入 Gateway
4. Gateway 根据路由目录转发到 IM Service
5. IM Service 写入 `workpal_im`
6. IM Service 发布 Redis Streams 事件
7. Search Service 消费事件并更新 Bleve
8. WebSocket 把消息推给在线成员

如果你能把这条链路讲清楚，这个项目最难的一层你就过了。

## 9. 当前项目最值得学习的地方

### 9.1 它已经摆脱了“假微服务”

它不是只分目录，而是：

- 服务独立启动
- 数据库按服务归属
- Gateway 是真实入口
- 搜索链路是异步解耦

### 9.2 它把微服务治理的基础味道做出来了

尤其是 Gateway 这一层，你能同时看到：

- 路由目录
- 流量治理
- 重试
- 熔断
- 健康检查
- 管理面

这对学习非常有价值。

### 9.3 它很适合从 Spring Cloud Alibaba 迁移认知

你可以把它当作：

- Go 语言生态下的轻量微服务课堂
- 用较少基础设施，演示较完整思路的样本

## 10. 还有哪些地方值得继续增强

### 10.1 当前还没有真正的动态注册中心

现在的服务目录是静态配置驱动的，这对本地学习很好，但离完整 Nacos 还差：

- 动态注册
- 实例摘除
- 配置热更新

### 10.2 当前 Gateway 仍然是单实例入口

它已经有治理能力，但还没有继续演进到：

- 多实例入口
- 更细粒度的租户 / 路由级限流
- 更丰富的追踪指标

### 10.3 WebSocket 还不是分布式连接层

IM Service 的连接管理仍然在单进程内存里。

## 11. 推荐练习题

如果你想继续深挖微服务部分，推荐按这个顺序练：

1. 给 Gateway 加上更丰富的 Prometheus 指标
2. 把 Gateway 的服务目录从静态配置升级为可热更新
3. 给内部 HTTP 客户端统一超时和重试策略
4. 给 IM Service 的 WebSocket 设计跨实例广播
5. 给 Search Service 增加全量重建索引命令

## 12. 最后给你的阅读建议

不要试图一口气把所有文件扫完。

更有效的方式是：

1. 选一条真实业务链路
2. 顺着链路看前端、网关、服务、存储
3. 每跨一层就停下来总结“这一层为什么存在”

把 Gateway 这一层看透之后，你读这个项目会轻松很多；因为后面的各个服务，其实都会回到边界、契约、存储、调用方式这四个问题上。
