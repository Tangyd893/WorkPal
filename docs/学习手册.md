# WorkPal 学习手册

这份手册面向"想通过一个完整项目学习微服务、网关治理、实时协作和前后端边界"的读者。它不追求一次讲完全部代码，而是帮你建立稳定的阅读顺序和判断框架。

## 1. 先用一句话看懂项目

> WorkPal 是一个由"前端工作台 + API Gateway + 多个领域微服务 + Redis 驱动的基础设施能力"组成的办公协作平台。

当前最值得学习的，不是单个 CRUD，而是这些能力如何拼在一起：

- Gateway 作为统一入口，承担路由、限流、重试、熔断、服务发现
- Redis 同时承担服务注册发现、IM 扇出、Redis Streams 异步链路
- 各服务各管各的数据边界，各管各的数据库
- 前端工作台通过网关整合多个领域能力
- 消息支持编辑、撤回、行内编辑
- 版本化数据库迁移体系覆盖四个领域服务

## 2. 推荐阅读顺序

建议按下面顺序走：

1. 先把项目跑起来
2. 看前端工作台有哪些模块
3. 看前端 API 是怎样只对接网关的
4. 把 Gateway 看透
5. 再看服务注册发现
6. 再看 IM 的跨实例实时链路
7. 再看 IM 到 Search 的 outbox 链路
8. 再看数据库迁移与工程化
9. 最后回到具体领域服务

这样读的好处是：你先理解系统为什么这样活着，再理解每个文件为什么这样写。

## 3. 用 Spring Cloud Alibaba 视角来映射

这不是把 Java 组件名字硬搬到 Go 里，而是借助成熟微服务视角来读当前实现。

| 学习视角 | 当前 WorkPal 实现 | 你应该重点看什么 |
| --- | --- | --- |
| Spring Cloud Gateway | `backend/cmd/gateway` | 路由、限流、重试、熔断、健康检查 |
| Nacos | `backend/internal/platform/registry.go` + `/gateway/services` | 服务注册、服务发现、静态回退 |
| Sentinel | Gateway 的限流、超时、重试、熔断、就绪检查 | 入口层治理策略 |
| Feign | `backend/internal/clients/*` | 服务间 HTTP 调用封装 |
| RocketMQ | `message_outbox` + `backend/pkg/msgqueue/redis_streams.go` | IM 到 Search 的可靠异步索引链路 |

如果你学的是"Spring Cloud 五大组件"那条路线，也可以这样理解：

- Gateway -> WorkPal Gateway
- Nacos -> Redis 服务注册发现
- Ribbon -> Gateway 基于发现结果做实例选择
- Feign -> `internal/clients`
- Sentinel / Hystrix -> Gateway breaker / retry / rate limit
- RocketMQ -> Outbox + Redis Streams

## 8. 第五阶段：看 IM 的跨实例实时能力

重点看：

- [cluster.go](/D:/Code/codex/WorkPal/backend/internal/im/ws/cluster.go)
- [hub.go](/D:/Code/codex/WorkPal/backend/internal/im/ws/hub.go)
- [message_handler.go](/D:/Code/codex/WorkPal/backend/internal/im/handler/message_handler.go)
- [cluster_test.go](/D:/Code/codex/WorkPal/backend/internal/im/ws/cluster_test.go)

你应该看懂两条链路。

### 8.1 用户定向消息链路

1. IM Service 处理消息写入
2. Handler 拿到会话成员列表
3. `ClusterBroadcaster.BroadcastUsers` 先投递本机在线用户
4. 再把事件发到 Redis Pub/Sub
5. 其他实例订阅后把消息推给各自本机在线用户

### 8.2 房间广播链路

1. WebSocket 客户端产生已读或已读全部等房间事件
2. Hub 先做本机房间广播
3. 再通过 cluster broadcaster 把房间事件扇出到其他实例
4. 其他实例对本机已加入该房间的客户端继续广播

### 8.3 消息编辑与撤回

前端新支持的消息操作：

1. 点击自己消息旁的"编辑"按钮进入行内编辑模式
2. 按 Enter 提交编辑，按 Escape 取消
3. 后端 `PUT /api/v1/messages/:id` 更新消息内容
4. 点击"撤回"按钮调用 `DELETE /api/v1/messages/:id`
5. 前端收到编辑/撤回的 API 响应后实时更新界面
6. IM 服务在消息变更时同步更新 `message_outbox`，触发搜索索引刷新

## 10. 第七阶段：看数据库迁移与工程化

这是 P1 优化中新增的工程化基础设施。

重点看：

- [migrations/](/D:/Code/codex/WorkPal/backend/migrations/) 目录下的四个服务迁移文件
- [Makefile](/D:/Code/codex/WorkPal/backend/Makefile) 中的 `migrate-*` 目标

你要看懂的关键点：

### 10.1 为什么需要版本化迁移

之前项目依赖 GORM 的 `AutoMigrate` 在启动时自动建表。这在开发阶段很方便，但在真实项目中存在几个问题：

1. **不可回溯：** 你无法知道数据库经过了哪些变更
2. **不可回滚：** 出问题时无法快速回到上一版本
3. **不可审计：** 团队协作时无法对 schema 变更做 code review
4. **不精确：** AutoMigrate 只能添加列，不能做复杂的迁移操作

### 10.2 golang-migrate 的使用方式

```powershell
# 安装 CLI 工具
make migrate-install

# 运行所有未执行的迁移
make migrate-up SERVICE=user-service

# 回滚最近一次迁移
make migrate-down SERVICE=im-service

# 创建新的迁移文件
make migrate-create SERVICE=workspace-service NAME=add_task_labels
```

每个迁移包含两个 SQL 文件：

- `NNNN_name.up.sql` — 前滚（创建表、添加列等）
- `NNNN_name.down.sql` — 回滚（对应的 DROP 操作）

### 10.3 迁移与 AutoMigrate 的关系

当前项目两种方式都可以使用：

- **快速开发：** 直接启动服务，AutoMigrate 自动建表
- **正式部署：** 先运行迁移，再启动服务

建议在 CI 和正式环境中优先使用迁移，确保 schema 变更有迹可循。

## 13. 当前项目的学习价值与边界

### 已经很值得学的部分

- 微服务入口层治理（路由、限流、熔断、重试）
- 轻量服务注册发现（Redis + 静态回退）
- Redis 同时承担多类基础设施职责
- 实时系统从单实例向多实例演进（Pub/Sub 扇出）
- Outbox + Redis Streams 的最终一致性异步链路
- Repository 接口模式用于单元测试 mock
- 版本化数据库迁移体系

### 还没有完全做满的部分

- 还没有独立配置中心
- 还没有分布式链路追踪体系（OpenTelemetry）
- 服务发现仍是轻量版，不是完整注册中心
- 还没有 API 文档生成（OpenAPI/Swagger）

## 14. 你接下来最该带着的问题去读

1. 入口层为什么必须独立于业务服务
2. 服务注册发现和静态配置回退为什么可以同时存在
3. WebSocket 多实例为什么不能只靠本机内存 Hub
4. 为什么 Redis 在微服务项目里经常不止扮演缓存
5. 为什么"能跑"不等于"已经工程化完成"
6. 为什么数据库迁移要从 AutoMigrate 升级到版本化管理
7. 为什么 Repository 接口模式对单元测试很重要

如果你把这七个问题想透，这个项目对你学习微服务就已经很有价值了。
